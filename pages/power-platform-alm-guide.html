Section 2

The 6 Foundation Decisions That Determine Project Failure or Success

Structural correctness must be established before you write a single pipeline. Each decision below is a project-killer if deferred.

2a — Managed vs Unmanaged Solution Layers

Every Dataverse environment maintains two solution layers: managed (read-only, deployed from upstream) and unmanaged (editable, the developer's working layer). The managed layer is what ALM deploys. The unmanaged layer is what developers create in.

❌ Anti-Pattern — Unmanaged Layer Corruption Importing an unmanaged solution to Test/UAT/Prod creates an editable working layer in a downstream environment. When a managed solution is then deployed on top, the two layers conflict silently: the managed layer wins for reads, but the unmanaged layer persists and resurfaces when the managed solution is deleted or downgraded. This silently destroys ALM repeatability — environment state can no longer be explained by source control alone.
✅ Rule — Managed Solutions ONLY in Test / Stage / Prod Enforce this via pipeline service principal permissions: the service principal that deploys solutions should be the only identity with import rights in Test/Stage/Prod. Revoke maker import permissions on those environments. Use pac solution import --managed in all CD stages.

2b — Solution Segmentation Strategy

❌ Anti-Pattern — The Monolith Solution One solution containing all components (tables, apps, flows, PCF, connectors) creates unresolvable merge conflicts, impossible independent deployments, and cascading failures when any component breaks.
✅ Recommended — 3-Layer Solution Architecture 01-CoreData → 02-SharedComponents → 03-AppName. Each layer depends on the one above. Deploy in order; never deploy App before CoreData is stable.

2c — Publisher Prefix and Naming Conventions

❌ Anti-Pattern — Using the Default new_ Publisher The default publisher prefix new_ is shared across all environments, all tenants, and all solutions. It cannot be changed after a component is created without recreating the component. Using it in enterprise projects makes component ownership impossible to determine and creates naming collisions.

Publisher prefix must be agreed and locked before any component is created. Recommended format: [clientcode] — 3–5 lowercase letters, no underscores in the prefix itself. Example: contoso → components will be contoso_tablename, contoso_columnname.

💡 Licensing Precision — End Users in Managed Environments Users of apps in a Managed Environment do not need a separate Managed Environment licence — that applies to the environment itself, not users. However: if the app uses premium connectors (SharePoint via custom connector, Dataverse, etc.), each end user needs Power Apps Premium (or per-app plan). Standard/seeded scenarios (Office 365 connector, basic SharePoint, Teams) do not require premium for end users.
⚠️ Licensing is Subject to Microsoft Change Validate all licensing assumptions with your Microsoft account team before project start. Microsoft has changed licensing boundaries multiple times — assumptions made at proposal stage may not hold at go-live.

Which environments should be Managed? Target stages for PP Pipelines (Test, Stage, Prod) must be Managed. Individual developer environments typically do not need to be Managed — this avoids per-developer overhead and allows unrestricted experimentation.

2f — Naming and Governance Baseline Checklist

All 10 items must be in place before any pipeline is created:

  • ☐ Publisher prefix agreed, documented, and locked — all team members know it, no exceptions
  • ☐ Solution segmentation strategy defined — at least CoreData / SharedComponents / App layers agreed
  • ☐ Naming conventions table published and distributed (solution, env var, connection ref, flow, app)
  • ☐ Individual dev environments provisioned — one per developer, no shared dev environment
  • ☐ Target environments (Test, Stage, Prod) created as Managed Environments if PP Pipelines is the toolchain
  • ☐ Service principal (app registration) created for pipeline deployments — System Administrator scoped per environment
  • ☐ DLP policies applied to all environments — including individual dev environments
  • ☐ Security groups created and linked to each environment — no direct user assignment to environments
  • ☐ Source control repository initialised with solution unpack structure — at least one baseline commit of each solution in its starting state
  • ☐ Component ownership matrix agreed — who owns which form, app, flow (prevents parallel edits on conflict-prone components)

Pipeline Host Environment (PP Pipelines only)

When using Power Platform Pipelines, a dedicated host environment must be provisioned. This environment: installs the Deployment Pipeline Configuration app, defines pipeline records (stage order, approvers, delegated deployment settings), must be Managed, and must be separate from all deployment target environments. The host environment does not receive solution deployments — it only orchestrates them. Typically owned by the Platform Admin team with strictly controlled access.

Build/Validation Environment (ADO only)

💡 Build Environment Design The build environment exists solely for the CI pipeline service principal. It is used to: (1) export the latest solution state if building from environment rather than source, (2) run solution checker via PAC CLI, (3) validate that the unpacked source compiles into a valid managed solution. No human should have maker access to this environment — its state is entirely controlled by the pipeline. Refresh it periodically (delete + recreate) to avoid accumulated state drift.

Region and Data Residency

All environments in the chain should be in the same Azure geography as Production unless there is a specific regulatory reason (e.g., UAT in a different region for data isolation testing). Cross-region deployments add latency to pipeline runs and can cause connector availability differences. For EU data residency requirements: ensure all environments — including individual dev environments — are provisioned in the EU geography from day one. Migrating environments across geographies is not supported; it requires environment recreation.

Section 4

How 10 Developers Merge Without Breaking Each Other

💡 Core Principle Source control is the single source of truth — not environments. If it is not in git, it does not exist. Environments are ephemeral; git history is permanent.
❌ Anti-Pattern — "Sync Environments" as a Merge Strategy "Just copy from Dev-3 to Test" is not ALM. Environment sync bypasses code review, loses history, cannot be rolled back, cannot be diffed, and breaks the moment two developers make conflicting changes. The only safe promotion path is: source control → pipeline artifact → managed deployment.

Unpacked Solutions in Git

pac solution unpack --zipfile MySolution.zip --folder ./src/MySolution --processCanvasApps decomposes the solution zip into individual files per component: one XML file per table, one JSON per flow, one YAML per canvas screen. This enables:

  • Line-level diffs in PRs — reviewers can see exactly what changed in a form, flow, or app
  • Merge conflict resolution at the component level instead of binary zip conflicts
  • Solution checker as a PR gate (run against unpacked source)
  • pac solution pack --folder ./src/MySolution --zipfile MySolution_managed.zip --managed rebuilds the deployable artifact from source
⚠️ Always Use --processCanvasApps Without this flag, canvas apps remain as a single binary MSAPP file — undiffable and unresolvable in merge conflicts. Always unpack with this flag; the YAML-per-screen format is the only mergeable form.

For 10-dev teams with canvas apps and model-driven forms, GitFlow Lite is safer.

Day in the Life — Developer

  1. Pull latest develop: git pull origin develop
  2. Create feature branch: git checkout -b feature/[ticket]-[short-desc]
  3. Make changes in personal dev environment (Dev-[name])
  4. Export solution: pac solution export --name MySolution --path ./MySolution.zip --managed false
  5. Unpack: pac solution unpack --zipfile MySolution.zip --folder ./src/MySolution --processCanvasApps
  6. Review diff: git diff — confirm only expected files changed
  7. Commit and push: git add src/MySolution/ && git commit -m "feat([ticket]): description" && git push origin feature/[ticket]-[short-desc]
  8. Open PR to develop — link ticket, describe what changed and why
  9. Address review comments, re-export if changes needed in dev env

Day in the Life — Release Manager

  1. Confirm all feature PRs for the sprint are merged to develop
  2. Create release branch: git checkout -b release/vX.Y origin/develop
  3. Bump solution version in solution.xml: Version="X.Y.Z.build"
  4. PR release branch to main — triggers build pipeline
  5. Build pipeline: pack managed artifact → publish to ADO Artifacts / pipeline run artifact
  6. CD pipeline Stage 1: deploy to Test — run automated post-deploy checks — approve or reject
  7. CD pipeline Stage 2: manual approval gate → deploy to Stage → smoke test checklist
  8. CD pipeline Stage 3: Change Board approval → deploy to Prod → post-deploy verification → tag release: git tag v1.2.0
  9. Backport release branch to develop: git merge release/vX.Y

Hotfix Strategy

StepAction
1. Branch from maingit checkout -b hotfix/[desc] main — NOT from develop (avoid pulling in unreleased sprint work)
2. Make fixExport from production-mirroring environment, unpack, commit
3. Fast-track PRSenior dev review only — abbreviated checklist, no full UAT cycle

Component Ownership Model

Assign ownership before sprint planning, not during. Each conflict-prone component (canvas app, MDA form, sitemap, ribbon) has exactly one owner at any time. Ownership can transfer between sprints via explicit handoff. Two developers should never be in the same component in the same sprint without a coordination protocol (e.g., one owns screens A–C, other owns screens D–F with no cross-screen formula dependencies).

Section 5

Azure DevOps ALM: Setup to Production

5a — Prerequisites & Service Principal Setup

  1. App registration in Entra ID — create a new app registration (no redirect URI needed). Note the Application (client) ID and Tenant ID. Generate a client secret (store in Key Vault — never in pipeline YAML).
  2. Dataverse application user — in each target environment (Build, Test, Stage, Prod): Power Platform Admin Centre → Settings → Users → Application Users → New → paste the App ID. Assign role: System Administrator. Scope this per-environment — the SP should have no higher org-level role than needed.
  3. ADO service connection — in ADO Project Settings → Service Connections → New → Power Platform type. Enter: Server URL (environment URL), Tenant ID, Application ID, Client Secret. Test connection before saving. Create one service connection per environment (not one shared connection for all).
  4. Variable groups — create one variable group per environment (e.g., vg-pp-test, vg-pp-prod) containing: ServiceConnectionName, EnvironmentUrl, SolutionName. Link to Azure Key Vault for secrets — client secrets and API keys must never be stored as plain ADO variables.
  5. PAC CLI vs Power Platform Build Tools — use PAC CLI tasks (microsoft/powerplatform-actions or inline pac commands via script task). Build Tools tasks exist but are less flexible and lag behind PAC CLI features. If using Build Tools, note: Pack Solution = pac solution pack, Import Solution = pac solution import.
  6. Agent pool — Microsoft-hosted (ubuntu-latest or windows-latest) is sufficient for most projects. Use self-hosted when: private network Dataverse access is required, custom tooling (specific PAC CLI version, internal scripts) must be pre-installed, or pipeline run minutes are a cost concern at scale.

5d — CD Pipeline Design (Multi-Stage)

Deployment settings files (config/[stage]/deployment-settings.json) inject environment-specific values for connection references and environment variables at deploy time — the same managed artifact deploys everywhere; only the settings file changes.

CD StageStepsApprovalOn failure
Stage 1 — TestDownload artifact → pac solution import --managed --deploymentSettingsFile config/test/deployment-settings.json --activatePlugins → run post-deploy-validate.ps1 (env var check, connection ref status, flow state)Auto-approve (or 1-hour window for QA team to reject)Pipeline fails → notify QA lead → artifact stays in Test for investigation
Stage 2 — StageManual approval gate (Release Manager) → deploy with stage deployment settings → smoke test checklist validationRelease Manager must approve — 48h timeoutReject → investigation → fix in next build — no rollback unless QA sign-off

💡 Deployment Settings — Connection Reference & Env Var Injection The deployment-settings.json file maps connection reference logical names to pre-created connection IDs in each environment, and maps environment variable schema names to their target values. Without this, flows and apps break after deployment because they cannot resolve connections to environment-specific resources. Generate the template once with pac solution create-settings --solution-zip MySolution.zip --settings-file deployment-settings.json then fill in the values per environment.

5e — YAML Pipeline Examples

📄 Example 1 — PR Validation Pipeline (solution checker gate) 
# ci-pr-validate.yml — triggers on PR to develop
trigger: none
pr:
  branches:
    include: [develop, release/*]

pool:
  vmImage: ubuntu-latest

variables:
  - group: vg-pp-build   # contains SolutionName, BuildEnvUrl

steps:
  - task: PowerShell@2
    displayName: Install PAC CLI
    inputs:
      targetType: inline
      script: dotnet tool install --global Microsoft.PowerApps.CLI.Tool
      # Build Tools equivalent: use PowerPlatformToolInstaller task

  - task: PowerShell@2
    displayName: Pack solution from source
    inputs:
      targetType: inline
      script: |
        pac solution pack `
          --folder $(Build.SourcesDirectory)/solutions/$(SolutionName)/src `
          --zipfile $(Build.ArtifactStagingDirectory)/$(SolutionName).zip

  - task: PowerShell@2
    displayName: Run Solution Checker
    inputs:
      targetType: inline
      script: |
        pac solution check `
          --path $(Build.ArtifactStagingDirectory)/$(SolutionName).zip `
          --outputDirectory $(Build.ArtifactStagingDirectory)/checker `
          --geo Europe
        # Parse results — fail if High or Critical violations found
        $results = Get-Content "$(Build.ArtifactStagingDirectory)/checker/*.sarif" | ConvertFrom-Json
        $critical = $results.runs.results | Where-Object { $_.level -in @('error','warning') -and $_.properties.severity -in @('High','Critical') }
        if ($critical.Count -gt 0) { Write-Error "Solution Checker: $($critical.Count) High/Critical violations"; exit 1 }
      # Build Tools equivalent: PowerPlatformCheckerTask

  - task: PublishTestResults@2
    displayName: Publish checker results
    inputs:
      testResultsFormat: NUnit
      testResultsFiles: '**/checker/*.xml'
      failTaskOnFailedTests: true

Power Platform Pipelines: Setup, Strengths, and Gaps

Power Platform Pipelines is Microsoft's built-in deployment orchestration for Power Platform — no ADO, no YAML, no CLI. Designed for maker-led teams and low-code environments. Target audience: citizen developers, small pro-dev teams, and organisations without DevOps capability.

Step-by-Step Setup

  1. Provision host environment — create a dedicated Sandbox environment (e.g., Contoso-PipelineHost). This must be a Managed Environment. It will not receive solution deployments — it only hosts pipeline configuration.
  2. Install Deployment Pipeline Configuration app — in the host environment: Power Platform Admin Centre → Dynamics 365 apps → Install Power Platform Pipelines. Alternatively, install via pac application install --environment [host-env-url] --application-name "Deployment Pipeline Configuration".
  3. Link target environments — in the Pipeline Configuration app, create Deployment Environment records for each stage (Test, Stage, Prod). Each target must be a Managed Environment. The service connection between host and targets is established via system-to-system trust — no additional SP setup required for basic pipelines.
  4. Create pipeline record — New Pipeline → name it → assign the linked deployment environments in order (Test → Stage → Prod). Set deployment order and whether each stage is sequential.
  5. Configure approval gates — per stage: enable Pre-deployment approval → assign approvers (individuals or Dataverse teams). Approvers receive email and can approve/reject via the Power Platform maker portal.
  6. Configure delegated deployment (optional) — allows makers to trigger deployments without needing system admin access to target environments. The pipeline service account handles the import; the maker only initiates the run.
  7. Trigger a deployment — from a solution in the source (developer) environment: Solutions → select solution → Pipelines → Deploy here → runs to Test → awaits approval → Stage → Prod.

Environment Variable & Connection Reference Handling

When a pipeline deployment reaches a new environment, Power Platform Pipelines automatically detects unresolved connection references and environment variables. The deploying user (or approver) is prompted to configure them inline via the pipeline UI before the deployment proceeds. Values are stored per-environment in the Pipeline Configuration app. This is the key advantage over manual import — it guides configuration without requiring deployment settings files or script knowledge.

⚠️ Inline Configuration Has Limits The inline prompt only appears when a value is missing. If a connection reference was previously set to a stale or wrong connection, the pipeline will not re-prompt — it will reuse the stored value silently. Audit connection reference assignments after each deployment, especially after connector re-authentication events.

Redeploy a Prior Version

  1. Navigate to the Pipeline Configuration app in the host environment (not the maker portal)
  2. Open Pipeline Runs — find the prior successful run that contains the version to restore
  3. Select the run → Redeploy — this re-triggers the same artifact through the approval chain
  4. Complete approval gates as normal — the prior managed solution version is imported over the current one using stage-and-upgrade behaviour
💡 Retain Pipeline Run History Pipeline run history is stored in the host environment's Dataverse tables. Do not delete the host environment or pipeline records — doing so removes the ability to redeploy prior versions. Treat the host environment as permanent infrastructure.

Post-Deployment Validation (Automated)

Run immediately after every managed solution import in CI/CD. Catches the most common deployment failures before a human runs a smoke test.

📄 post-deploy-validate.ps1 — key checks 
param($EnvUrl, $AppId, $Secret, $TenantId, $SolutionName)

# Auth via service principal
pac auth create --url $EnvUrl --applicationId $AppId --clientSecret $Secret --tenant $TenantId

# 1. Check environment variables are all set (no null current values)
$envVars = pac env list-variables --environment $EnvUrl --json | ConvertFrom-Json
$unset = $envVars | Where-Object { $_.currentValue -eq $null -and $_.defaultValue -eq $null }
if ($unset.Count -gt 0) {
    Write-Error "UNSET ENV VARS: $($unset.schemaName -join ', ')"
    exit 1
}

# 2. Check connection references are all configured
$connRefs = pac env list-connection-references --environment $EnvUrl --json | ConvertFrom-Json
$unconfigured = $connRefs | Where-Object { $_.connectionId -eq $null }
if ($unconfigured.Count -gt 0) {
    Write-Error "UNCONFIGURED CONN REFS: $($unconfigured.logicalName -join ', ')"
    exit 1
}

# 3. Check all flows in solution are enabled (not suspended)
# Requires Dataverse API call - flows suspended due to connection issues fail here
$token = (pac auth token --environment $EnvUrl --json | ConvertFrom-Json).accessToken
$flows = Invoke-RestMethod -Uri "$EnvUrl/api/data/v9.2/workflows?`$filter=category eq 5 and statecode eq 0" `
  -Headers @{Authorization="Bearer $token"; "OData-MaxVersion"="4.0"}
$suspended = $flows.value | Where-Object { $_.statuscode -eq 2 }  # 2 = Suspended
if ($suspended.Count -gt 0) {
    Write-Warning "SUSPENDED FLOWS: $($suspended.name -join ', ')"
    # Warning only — flows may be intentionally off; human must verify
}

Write-Host "✅ Post-deploy validation passed"

Connection Reference Post-Deployment Configuration

Connection references are the mechanism by which flows and apps connect to external services without hardcoding connection details. After a managed solution is deployed to a new environment, each connection reference must be mapped to a valid connection in that environment — the connection from Dev does not follow the solution.

❌ What Happens Without Pre-Configuration If connection references are not pre-configured via deployment settings: all flows that use those connectors are suspended immediately on import. Canvas apps that call those flows or connectors show errors. The only fix is to manually open each flow, update the connection, and turn the flow back on — which is manual, error-prone, and untraceable.
✅ How Deployment Settings Files Solve This In config/[stage]/deployment-settings.json, map each connection reference logical name to a pre-created connection ID in that environment. The connection must exist before the pipeline runs. PAC CLI creates the connection in the environment; the ID is then stored in the settings file. During import, pac solution import --deploymentSettingsFile applies the mapping automatically — flows are enabled, apps are connected.

❌ Hardcoding Violations — Zero Tolerance in Enterprise Projects

Rollback: When It's Safe, When It's Dangerous, and How to Decide

⚠️ Core Message Rollback is a last resort. A forward hotfix is almost always safer after a destructive schema change. Rollback thinking should begin at design time — not at 2am during an incident.

Rollback Safety Decision Table

ScenarioRollback safe?RiskRecommended action
Flow logic change only (no schema, no data)✅ SafeMinimal — prior managed artifact restores flow definitionRedeploy prior artifact via ADO or PP Pipelines. Verify flows re-enabled after import.
App UI change only (no connector or formula changes)✅ SafeLow — prior app screens restoredRedeploy prior artifact. Re-publish the app after import if needed.
Schema change — column added⚠️ PartialColumn remains in environment after rollback (managed solution does not delete columns on downgrade when using update behaviour). Existing data in the column is preserved.Rollback the managed solution. If column is unwanted, remove it manually post-rollback via unmanaged change — then address in next managed release.
Schema change — column deleted❌ DangerousData in the deleted column is permanently lost. Rollback restores the solution definition but cannot restore data.Do NOT rollback. Hotfix forward: restore the column definition in source, redeploy as new version. Assess data loss with business; restore from Dataverse backup if within backup window (default: 28 days).
Data migration applied (bulk transform, delete, or re-keying)❌ DangerousSolution rollback does not reverse data changes. Data is in the state the migration left it.Do NOT rollback managed solution. Restore from Dataverse point-in-time backup if data loss is unacceptable. Treat as a data incident, not an ALM incident.
Managed solution layer corruption (unmanaged layer exists in downstream env)❌ Not possible via normal rollbackActive unmanaged layer overrides or conflicts with managed solution — both layers coexist and produce unpredictable behaviourRemove the unmanaged customisation manually (delete the unmanaged solution layer or specific components), then redeploy the managed solution. Root cause: an unmanaged import reached a downstream environment — enforce controls to prevent recurrence.

ADO Rollback — Redeploy Prior Artifact

  1. Identify the prior successful pipeline run containing the target version — find it in ADO Pipelines → Runs → filter by cd-deploy
  2. Note the artifact version (build number or git tag, e.g., v1.1.4)
  3. Re-trigger the CD pipeline from that specific run: Run → Rerun, or trigger a new CD run that downloads the prior artifact by build ID
  4. Preferred: use pac solution import with --stageAndUpgrade flag — this atomically replaces the current managed solution with the prior version
  5. Do not use --update for rollback — it leaves deleted components as unmanaged orphans in the environment
  6. Run post-deploy validation script after import completes
⚠️ stage-and-upgrade vs update on Rollback --stageAndUpgrade: imports as a holding solution, then upgrades atomically — components removed in the prior version are properly cleaned up. --update: applies delta only — any components that were added in the broken version but absent in the prior version remain as unmanaged orphans. Always use --stageAndUpgrade for rollback.

PP Pipelines Rollback — Exact UI Steps

  1. Open the Pipeline Configuration app in the host environment (not the maker portal — go to the host environment's model-driven app)
  2. Navigate to Pipeline Runs in the left navigation
  3. Filter by pipeline name and locate the prior successful run (check the Deployed On date and solution version number)
  4. Open the run record → click Redeploy in the command bar
  5. Complete the approval gates as normal — the redeployment goes through the full approval chain
  6. After redeployment, verify connection references and environment variables are still correctly configured (they should persist, but verify)

Decision factorFavour RollbackFavour Hotfix Forward

Data migration applied?No data changesYes — rollback cannot reverse data
Time to hotfix vs time to rollbackHotfix would take > 2 hoursHotfix is < 1 hour (simple logic fix)

Pre-Rollback Checklist (8 items)

  • ☐ Confirm prior artifact is available (ADO artifact store or PP Pipelines run history)
  • ☐ Assess data state — any data migrations or deletes that cannot be reversed?
  • ☐ Identify all dependent solutions — will rollback break downstream components?
  • ☐ Notify business stakeholders — set expected downtime window
  • ☐ Take a Dataverse point-in-time backup before rollback (Admin Centre → Environments → Backups → Restore)
  • ☐ Confirm approval chain is available (release manager, change board if required)
  • ☐ Prepare post-rollback validation script inputs (environment URL, SP credentials)
  • ☐ Document the decision: why rollback vs hotfix-forward, who approved, what data risk was assessed

Post-Rollback Verification Checklist (6 items)

  • ☐ Confirm solution version in environment matches expected prior version (pac solution list)
  • ☐ Run post-deploy validation script — all env vars set, all connection refs configured, flows enabled
  • ☐ Smoke test primary user journey end-to-end
  • ☐ Check for unmanaged orphan components — any components from the failed release still present as unmanaged layers?
  • ☐ Confirm Copilot Studio agents (if applicable) are operational and topics are responding correctly
  • ☐ Post incident review scheduled — root cause analysis, pipeline improvement action items

Power Automate Flows Triggered by Agents

When a Copilot Studio agent triggers a cloud flow action, that flow must be in the same solution as the agent (or a dependent solution already deployed). ALM considerations:

  • The flow's connection references must be configured in the target environment — same requirement as any other flow deployment
  • Flows triggered by agents often run in the agent's security context — ensure the agent's service account has appropriate Dataverse and connector permissions
  • Flow run history is separate from agent conversation history — monitor both when diagnosing issues post-deployment
  • If the flow is owned by the deployment service principal, re-assign ownership to a shared service account so operations teams can monitor and manage it (same pattern as Section 8)
