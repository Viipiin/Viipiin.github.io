What is the difference between Xrm.Page and formContext? Which should you use today?

Theory
▼

Xrm.Page is deprecated since D365 v9. formContext is the modern approach — derived via executionContext.getFormContext(). It is scoped to the specific form instance, making it reliable in multi-form and UCI scenarios.

Always accept executionContext as your first parameter and derive formContext from it. Never use Xrm.Page in new development.
function onLoad(executionContext) {
    var formContext = executionContext.getFormContext();
    // use formContext safely
}

How do you prevent a form save and show a custom validation message?

Practical
▼ 
function onSave(executionContext) {
    var formContext = executionContext.getFormContext();
    var saveEvent = executionContext.getEventArgs();
    var revenue = formContext.getAttribute("revenue").getValue();

    if (!revenue || revenue <= 0) {
        saveEvent.preventDefault();
        Xrm.Navigation.openAlertDialog({
            text: "Revenue must be a positive number.",
            title: "Validation Error"
        });
    }
}
Register on the form's OnSave event. Always call preventDefault() before showing the dialog.

Write a plugin that auto-sets a "Welcome Note" on a new Contact record.

Practical
▼ 
using Microsoft.Xrm.Sdk;
using System;

public class ContactCreatePlugin : IPlugin
{
    public void Execute(IServiceProvider serviceProvider)
    {
        var context = (IPluginExecutionContext)
            serviceProvider.GetService(typeof(IPluginExecutionContext));

        if (context.MessageName != "Create" || context.Stage != 20) return;
        if (context.PrimaryEntityName != "contact") return;

        var target = (Entity)context.InputParameters["Target"];
        target["description"] = $"Welcome! Record created on {DateTime.UtcNow:yyyy-MM-dd}.";
    }
}
Register at Stage 20 (Pre-Operation) so the value is part of the original Create — no extra Update needed, saving a DB round-trip.
5

What are Pre-Images and Post-Images? Why are they necessary?

Theory
▼

On an Update message, Target only contains changed fields. To read unchanged field values, you need a Pre-Image.

Entity preImage = context.PreEntityImages["ContactPreImage"];
string oldEmail = preImage.GetAttributeValue<string>("emailaddress1");

Entity target = (Entity)context.InputParameters["Target"];
// Use pre-image value if field wasn't changed in this update
string currentEmail = target.Contains("emailaddress1")
    ? target.GetAttributeValue<string>("emailaddress1")
    : oldEmail;
Register the Pre-Image in Plugin Registration Tool, specifying the alias (e.g., "ContactPreImage") and the specific fields you need. Do not request all fields — impacts performance.
6

How do you prevent an infinite plugin loop?

PracticalCommon Interview
▼

Three strategies:

1. Check Depth — if a plugin updates the same entity, D365 re-fires it. Guard against this:

if (context.Depth > 1) return;

2. Filtering Attributes — register the step to fire only when specific fields change. If your plugin doesn't write those fields, it won't re-fire.

3. Flag Field — set a boolean new_isprocessed = true and skip execution if already set.

What are the key steps to register a plugin using Plugin Registration Tool?

Practical
▼
  1. Connect PRT to your D365 environment
  2. Register Assembly — upload .dll, choose Isolation Mode = Sandbox
  3. Register Step:
    • Message: Create / Update / Delete
    • Primary Entity: account / contact / etc.
    • Stage: Pre-Validation (10), Pre-Op (20), or Post-Op (40)
    • Mode: Synchronous or Asynchronous
    • Filtering Attributes: specific fields only (best practice)
  4. Register Images if needed — Pre/Post with alias and field list
  5. Test in D365 and verify Plugin Trace Log
Always specify Filtering Attributes on Update steps — otherwise the plugin fires on every single field change, degrading performance.

How do you implement error handling in Power Automate (Try-Catch pattern)?

Practical
▼
  1. Wrap risky actions in a Scope named "Try"
  2. Add a second Scope named "Catch"
  3. Configure "Catch" scope Run After: check has failed, has timed out, is skipped
  4. Inside Catch: log to Dataverse table or send Teams/email notification
// Expression to get error message inside Catch scope:
result('Try_Scope')?['error']?['message']
The result() expression on the Try scope returns the full error object including message and code for logging.

What is a Child Flow and when do you use it?

Theory
▼

A Child Flow is an Instant Flow called by another flow via "Run a Child Flow." It accepts input parameters and returns outputs.

Use when:

  • Reusable logic needed across multiple parent flows (e.g., standard error notification block)
  • Breaking a large flow into manageable units
  • Shared business logic (e.g., "Create Case with defaults")
Child flows must be in the same environment and solution as parent flows for ALM portability.

What is a Business Process Flow (BPF) and when do you use it?

Theory
▼

A BPF is a guided UI process overlay that walks users through predefined stages and steps. Each stage can enforce required data before progressing.

Example — Sales process: Qualify → Develop → Propose → Close

Each stage has steps (required fields). Users cannot move to the next stage until steps are complete. BPF stores progress in its own Process entity table, enabling stage-level reporting.

3

Practical
▼

Yes. Control which BPF activates via:

  • Security Roles — assign BPF privileges to specific roles; users only see authorized BPFs
  • Order — higher order number = higher priority
  • JavaScript — programmatically switch the active process instance
formContext.data.process.setActiveProcessInstance(
    processInstanceId,
    function(status) { console.log("Switched: " + status); }
);
4

What is a Business Rule and when do you prefer it over JavaScript?

Theory
▼

Business Rules are no-code if-then logic on a form/entity that can set field values, show/hide fields, set required levels, show error messages, and lock/unlock fields.

Prefer Business Rules when: simple if-then logic, maintainable by admins without developer, or when server-side enforcement is needed (Entity scope).

Use JavaScript when: Web API calls, complex multi-condition loops, cross-form interactions, or advanced Xrm API features are needed.

How do you add a custom column to a system entity like Contact without breaking solutions?

What is Field-Level Security and how do you implement it?

Practical
▼

Field Security restricts access to specific columns, regardless of entity-level Security Role.

  1. Enable "Enable field security" on the column definition
  2. Create a Field Security Profile (Security → Field Security Profiles)
  3. Set permissions: Read / Update / Create = Allow or Restrict per field
  4. Assign the profile to Users or Teams
System Administrators bypass Field Security by default. It only restricts non-admin users.

What is Solution Layering and how does it affect customizations?

Theory
▼

When multiple solutions modify the same component, Dataverse maintains a layer stack. The topmost layer wins.

Layer order (top wins): Unmanaged customizations → Customer managed → ISV managed → Microsoft base

Use "See Solution Layers" in the maker portal to debug conflicts — it shows which solution is providing the active definition of any component.
3

How does CI/CD work for D365 using Azure DevOps?

Practical
▼
  1. Dev: Developer makes changes in DEV environment
  2. Export: Pipeline exports solution using Power Platform Build Tools Azure DevOps task
  3. Unpack: pac solution unpack converts .zip to source files (XML, JS, JSON) committed to Git
  4. PR Review: Team reviews XML/code diffs like any other codebase
  5. Build Pipeline: On merge — pac solution pack creates a managed .zip
  6. Release Pipeline: Imports managed solution to TEST → UAT → PROD with manual approvals between stages

What are Environment Variables in solutions and why are they important?

Theory
▼

Environment Variables store environment-specific configuration (API URLs, SharePoint paths, email addresses) that differ across DEV/TEST/PROD.

  • The same managed solution works across all environments without code changes
  • Values are set at import time via the solution import wizard
  • Flows and plugins reference the variable by schema name — not a hardcoded value
  • Supports String, Number, Boolean, JSON, and Data Source types
Never hardcode environment-specific URLs or keys in flows or plugins. Always use Environment Variables.

How do you query Dataverse using SQL?

Practical
▼

Dataverse exposes a TDS (SQL Server-compatible) endpoint. Connect via SSMS / Azure Data Studio:

  • Server: yourorg.crm.dynamics.com,5558
  • Authentication: Azure Active Directory
SELECT name, revenue, telephone1
FROM account
WHERE revenue > 1000000 AND statecode = 0
ORDER BY name;
TDS endpoint is SELECT only. No INSERT/UPDATE/DELETE. Use Web API, plugins, or flows for writes. Virtual tables and some metadata tables are not queryable.
2

What is FetchXML? Write a query to get top 10 open Opportunities over ₹5L.

Practical
▼

FetchXML is D365's native XML query language supporting aggregates, linked entities, and security-trimmed results.

<fetch top="10">
  <entity name="opportunity">
    <attribute name="name" />
    <attribute name="estimatedvalue" />
    <attribute name="ownerid" />
    <filter>
      <condition attribute="statecode" operator="eq" value="0" />
      <condition attribute="estimatedvalue" operator="gt" value="500000" />
    </filter>
    <order attribute="estimatedvalue" descending="true" />
  </entity>
</fetch>

Integration — Redshift, Dataflow & Azure

1

What is a Power Platform Dataflow and how does it work?

Theory
▼

A Dataflow is a self-service ETL (Extract, Transform, Load) tool using Power Query (M language):

  • Extract — from 100+ sources: SQL, Redshift, Excel, REST, SharePoint, etc.
  • Transform — filter, join, rename, type-cast via no-code UI or M formula
  • Load — into Dataverse standard or custom tables on schedule or on-demand

Data lands in Dataverse, fully queryable via Web API, FetchXML, or SQL endpoint.

▼
