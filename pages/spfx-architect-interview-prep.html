O 19.2
What SharePoint site templates are available OOTB and when do you choose each?
  • Team site (with M365 group) — collaboration focus; auto-creates Teams channel, Planner, shared mailbox. Best for project teams.
  • Team site (no M365 group) — standalone SharePoint site; no Teams/Planner. Best for intranet sub-sites, HR libraries, department pages.
  • Communication site — broadcast focus (one team publishes, everyone reads). Best for intranets, news hubs, policy sites. No M365 group.
  • Hub site — not a template but a designation. Associate team and communication sites to a hub to share navigation, branding, and News rollup across sites.
  • Document Center — legacy template for large document management; replaced in modern SP by standard document libraries with metadata.
O 19.3
What is the Highlighted Content web part and how does it differ from a List web part?
  • List web part — shows a specific configured view from a specific list/library on a specific site. Static source. Best for displaying live data from a known list.
  • Highlighted Content web part — uses the SharePoint Search index as its data source. Dynamically surfaces content matching a query (content type, tag, date range, author, keyword). Can aggregate content across all sites in the tenant.

Use Highlighted Content when: you want automatic rollup (e.g., "show all documents tagged 'Policy' created in the last 30 days across all sites"). The results update automatically as new content is indexed.

Limitation: search index has a crawl delay (typically minutes). New content won't appear instantly.

O 19.4
What are SharePoint site features and how do you activate/deactivate them?

Site features are toggleable capabilities bundled with SharePoint. They exist at two scopes:

  • Site collection features — Site Settings → Site collection features. Examples: SharePoint Server Publishing Infrastructure, Content Type Hub, Cross-Site Collection Publishing.
  • Site features — Site Settings → Manage site features. Examples: Team Collaboration Lists (creates default Announcements, Calendar, Tasks lists), SharePoint Server Publishing.

Via PowerShell:

# Activate a site feature
Enable-PnPFeature -Identity "87294c72-f260-42f3-a41b-981a2ffce37a" -Scope Site

# List all active features
Get-PnPFeature -Scope Web | Select-Object DisplayName, DefinitionId

Common gotcha: Activating Publishing Infrastructure on a modern site can break the modern page experience — some features conflict. Always test in dev first.

SharePoint Permissions — Levels, Unique & Groups

Permission management is one of the most common day-to-day responsibilities in SharePoint projects. These questions cover the full model — from default levels to broken inheritance and item-level permissions.
P 17.1
What are the default SharePoint permission levels and what can each do?

SharePoint ships with five default permission levels (plus two special ones):

  • Full Control — complete site administration, including creating/deleting lists, managing permissions, and activating features. Typically only site collection admins and owners.
  • Design — can create and modify pages, themes, style sheets, and master pages. Rarely used in modern SharePoint.
  • Edit — add, edit, delete items and lists. Cannot manage permissions or site settings. The standard contributor-plus level.
  • Contribute — add, edit, delete list items and documents. Cannot manage lists or views. Most team members land here.
  • Read — view-only access to all content.
  • View Only — like Read but prevents file downloads (useful for protected documents). Users can view in browser only.
  • Limited Access — automatically assigned when a user has access to a specific item but not the parent list/site. You cannot assign this manually.

You can also create custom permission levels by copying an existing level and adding/removing individual permissions (e.g., allow Add Items but remove Delete Items).

P 17.2
What is permission inheritance and when would you break it?

By default, every sub-site, list, library, and list item inherits permissions from its parent. Breaking inheritance ("unique permissions") creates an independent permission set for that object.

When to break inheritance:

  • A specific document library needs to be restricted to HR only while the rest of the site is accessible to all staff
  • Individual items in a list need different visibility (e.g., each employee sees only their own records)
  • A sub-site for a confidential project within a broader intranet

Risks of breaking inheritance excessively:

  • Performance — SharePoint has to check permissions at every object level; thousands of unique permissions on list items causes serious throttling and slow page loads
  • Maintenance complexity — permission changes at the parent no longer cascade down
  • Migration complexity — tools like ShareGate must map item-level permissions individually

Rule of thumb: break at the list/library level only. Avoid item-level unique permissions unless it's a genuine business requirement — always explore column-based audience targeting or Microsoft 365 groups first.

P 17.3
What are SharePoint groups vs Microsoft 365 groups and which should you use for site permissions?
  • SharePoint groups — exist within a single site collection. Managed from Site Settings → People and Groups. No Azure AD backing. Members are assigned a SharePoint permission level. Best for site-scoped access control.
  • Microsoft 365 groups — Azure AD-backed groups. Assigning an M365 group to a SharePoint site gives all members access and creates a connected experience (Teams channel, Planner, shared mailbox). Permissions are managed in Azure AD, not SharePoint.
  • Azure AD Security groups — no M365 connected experience. Can be nested. Best for large org-wide access control patterns managed by IT.

Best practice for new projects: Prefer M365 groups for team sites (they provision a Teams channel automatically). Use SharePoint groups for classic intranet scenarios or when you need granular sub-site control. Never add individual users directly to site permissions — always via groups.

P 17.4
How do you set unique permissions on a list item programmatically using REST API or PnP JS?
// PnP JS — break item inheritance and grant role
import { spfi, SPFx } from "@pnp/sp";
import { "@pnp/sp/security" } from "@pnp/sp/security";

const sp = spfi().using(SPFx(context));

const item = sp.web.lists.getByTitle("Projects").items.getById(42);

// Step 1: Break inheritance (true = copy existing permissions)
await item.breakRoleInheritance(true, false);

// Step 2: Get the role definition (e.g. "Contribute")
const roleId = await sp.web.roleDefinitions.getByName("Contribute").select("Id")();

// Step 3: Get the principal (user or group)
const user = await sp.web.siteUsers.getByEmail("user@contoso.com")();

// Step 4: Assign role
await item.roleAssignments.add(user.Id, roleId.Id);

Warning: break inheritance on items sparingly — each unique item permission is a separate security object in SQL. Lists with 50,000+ uniquely-permissioned items have caused tenant-wide performance degradation in production.

P 17.5
How do you audit permissions across a site collection — what tools do you use?
  • SharePoint Admin Center → Active sites → select site → Permissions — gives a top-level view
  • PnP PowerShell — Get-PnPListItemPermission, Get-PnPRoleAssignment for scripted audits across all lists and items
  • ShareGate — has a dedicated "Permissions Report" that exports all broken inheritance, group memberships, and user access to Excel — much faster than scripting for large sites
  • Microsoft 365 Admin Center → Reports → SharePoint activity — for access activity, not just assignments
# PnP PowerShell — export all lists with unique permissions
Connect-PnPOnline -Url "https://contoso.sharepoint.com/sites/hr" -Interactive
$lists = Get-PnPList
foreach ($list in $lists) {
  $items = Get-PnPListItem -List $list -Fields "Id","Title"
  foreach ($item in $items) {
    $perms = Get-PnPListItemPermission -List $list -Identity $item.Id
    if ($perms.HasUniqueRoleAssignments) {
      Write-Host "List: $($list.Title) | Item: $($item.Id) | UNIQUE PERMISSIONS"
    }
  }
}

Lists and Libraries are the foundation of SharePoint. A complete understanding means knowing not just how to use them but where they break down — thresholds, column limits, versioning costs, and metadata constraints.

L 18.2
What are the key SharePoint Online limits for lists and libraries you must know?
  • List view threshold: 5,000 items — any query returning more than 5,000 items at once is blocked. Applies to filters, sorts, and indexed column queries.
  • Maximum items in a list: 30 million items (soft technical limit; performance degrades well before this)
  • Maximum columns per list: 2,048 columns of any type combined; but practical limit is much lower — each indexed column adds query overhead
  • Maximum indexed columns: 20 per list
  • Document library size: 25 TB per library (site collection quota usually limits this first)
  • Max file size: 250 GB per file
  • Attachment size on list items: 250 MB per attachment
  • File name length: 400 characters including path; avoid special characters # % & * : < > ? / \ { | } ~
  • Versioning storage: each version stores a full copy of the file — a 10MB document with 100 versions uses 1GB. Enable version limits.
  • Site collection storage: 25 TB per site collection in SPO
  • Lookup columns: max 12 lookup columns per view (adds a SQL join per lookup)
L 18.3
How do you work around the 5,000 item list view threshold?

Four strategies, in order of preference:

  1. Indexed columns + filtered views — add an index on the column you filter/sort by. Every view that returns <5,000 items by using that indexed column avoids the threshold. Do this before the list exceeds 5,000 items — adding an index to a large list triggers a full scan and may time out.
  2. PnP JS server-side paging — use getPaged() rather than loading all items at once: 
    let page = await sp.web.lists.getByTitle("Requests").items
  .select("Id", "Title", "Status")
  .filter("Status eq 'Open'")
  .top(500)
  .getPaged<IItem[]>();

while (page.hasNext) {
  page = await page.getNext();
  // process page.results
}
  3. Folders for partitioning — folders are transparent to the threshold. A library with 500K files across 100 folders can serve each folder's items without hitting the limit.
  4. Microsoft Search or Graph Search — for read-only lookup scenarios, use the Search API which has no threshold. Results come from the search index, not direct SQL.

L 18.5
How does versioning work in SharePoint and what storage impact should you plan for?

Document Libraries:

  • Major versions (1.0, 2.0, 3.0) — published, visible to all readers
  • Minor versions (1.1, 1.2) — draft state, visible only to contributors and above. Enable minor versioning when you need a review/approval workflow before publishing.
  • Version limits — set "Keep the following number of major versions" to prevent unbounded growth. Recommended: 10–20 major + 5 minor for most document libraries.

Storage impact: SharePoint does not store deltas — each version is a full copy. A 5 MB PowerPoint with 50 versions uses 250 MB. Multiply this across a 10,000-document library and storage costs escalate fast.

Best practice: Always set version limits. Use a Power Automate flow to archive old versions to Azure Blob Storage if long-term retention is required without storage cost.

N 20.1
What role does Node.js play in SPFx development and which versions are supported?

Node.js is the build-time runtime for SPFx — it powers the toolchain (Gulp, Webpack, Yeoman) but does not run in the browser. The browser only receives the compiled JavaScript bundle.

N 20.2
What does the SPFx build chain look like — what do Gulp and Webpack each do?
  • Yeoman (@microsoft/generator-sharepoint) — scaffolds the project structure, creating all config files, package.json, tsconfig, gulpfile, and the initial web part class.
  • npm — installs all node modules declared in package.json including the SPFx build toolchain (@microsoft/sp-build-web, @microsoft/sp-webpart-workbench).
  • Gulp — the task runner. Key tasks:
    • gulp serve — local development server with hot reload on the SPFx workbench
    • gulp bundle --ship — production bundle (minified, no source maps)
    • gulp package-solution --ship — creates the .sppkg file for App Catalog upload
  • Webpack — bundler called internally by Gulp. Combines all TypeScript/React code + imports into a single JS bundle per web part. Handles tree-shaking, code splitting, and externals (libraries loaded from CDN instead of bundled).
  • TypeScript compiler (tsc) — transpiles TypeScript to JavaScript. Configuration in tsconfig.json.
N 20.3
How do you implement responsive design in an SPFx web part?

Three complementary approaches:

  1. Fluent UI Stack + responsive breakpoints — Fluent UI's Stack component is a flexbox wrapper. Combine with StackItem and conditional props based on screen width.
  2. CSS modules with media queries — SPFx uses CSS modules (scoped class names). Write breakpoints in the .module.scss file: 
    /* Component.module.scss */
.card {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  gap: 16px;

  @media (max-width: 768px) {
    grid-template-columns: 1fr;
  }
}

.title {
  font-size: 18px;

  @media (max-width: 480px) {
    font-size: 14px;
  }
}
  3. useWindowDimensions hook — for layout changes that must happen in React state (e.g., switching between a table and a card layout): 
    function useWindowWidth() {
  const [width, setWidth] = useState(window.innerWidth);
  useEffect(() => {
    const handler = () => setWidth(window.innerWidth);
    window.addEventListener("resize", handler);
    return () => window.removeEventListener("resize", handler);
  }, []);
  return width;
}

SPFx-specific tip: The web part container width is available via this.domElement.clientWidth in the web part class — pass it as a prop to React for layout decisions based on the web part's actual width, not the browser window width (web parts can be in 1/3 column layouts).

G 21.2
What branching strategy do you use for SharePoint/SPFx projects?

I follow a simplified GitFlow adapted for SharePoint projects:

  • main — production-ready code only. Protected branch. Merges via PR only after pipeline passes.
  • develop — integration branch. All feature branches merge here first. Maps to UAT/Staging environment in the pipeline.
  • feature/[ticket-id]-short-description — one branch per feature or bug fix. E.g., feature/SP-42-paging-large-lists. Created from develop, merged back to develop via PR.
  • hotfix/[description] — emergency fixes branched from main, merged to both main and develop after resolution.

Branch policies in Azure DevOps (mandatory for any team project):

  • Require minimum 1 reviewer approval on PRs to develop/main
  • Require linked work item (ties code to Azure Boards tickets)
  • Require build to pass before merge (builds must be green)
  • Automatically delete source branch after merge
G 21.3
How do you resolve a merge conflict in Git?
# 1. Start the merge or rebase that causes the conflict
git merge develop

# 2. Git marks conflicts in the file with conflict markers:
<<<<<<< HEAD
  // Your local version
=======
  // Incoming version from develop
>>>>>>> develop

# 3. Edit the file: keep what's correct (or combine both), remove the markers

# 4. Stage the resolved file
git add src/webparts/MyWebPart/MyWebPart.ts

# 5. Complete the merge
git merge --continue
# or for rebase:
git rebase --continue

Tools: VS Code has an inline conflict editor (Accept Current / Accept Incoming / Accept Both) that removes the need to manually edit conflict markers. Azure DevOps PR interface also shows conflicts inline for simple resolutions.

G 21.5
What is a Pull Request and why is it mandatory on team projects?

A Pull Request (PR) is a request to merge a feature branch into a target branch (develop or main). It is the code review gate — no code reaches the main codebase without at least one reviewer approving it.

What a good PR process enforces:

  • Code review — another developer reads the diff and catches bugs, security issues, or design problems before they ship
  • Build validation — the CI pipeline must pass (lint, tests, build) before merge is allowed
  • Linked work items — every PR traces back to a ticket in Azure Boards, creating a full audit trail
  • Discussion thread — comments on specific lines; the author must resolve all comments before merge

In Azure DevOps: set branch policies on develop and main to require PR + 1 reviewer + passing build. This cannot be bypassed even by admins (unless you explicitly grant the bypass permission — which you shouldn't).

Throttling & Error Handling Cheat Sheet

// SharePoint throttle codes: 429 (Too Many Requests), 503 (Service Unavailable)
// Retry-After header: seconds to wait before retry

// Recommended retry pattern (from Section 09):
async function withRetry<T>(
  fn: () => Promise<T>,
  maxAttempts = 4,
  baseDelayMs = 1000
): Promise<T> {
  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    try {
      return await fn();
    } catch (err: unknown) {
      const e = err as { status?: number; headers?: Headers };
      if (attempt === maxAttempts) throw err;
      const delay = e.status === 429
        ? (parseInt(e.headers?.get("Retry-After") ?? "5") * 1000)
        : baseDelayMs * 2 ** (attempt - 1);
      await new Promise(r => setTimeout(r, delay));
    }
  }
  throw new Error("Unreachable");
}

Power Automate — Patterns to Know Cold

  • Try/Catch/Finally: three parallel scopes after a critical action; Catch scope has "Run after: Has failed"; Finally scope has "Run after: Is skipped, Has failed, Has timed out, Succeeded"
  • Retry policy: set on individual actions → Settings → Retry Policy → Exponential Interval (count 4, interval PT5S, max interval PT1H)
  • Concurrency control: Apply to each → Settings → Concurrency Control → On, degree = 1 (sequential) or up to 50 (parallel, watch for throttling)
  • Solution-aware flows: created inside a Dataverse solution; use connection references, not hardcoded credentials; transportable via managed/unmanaged solutions
  • Child flows: use "Run a Child Flow" action; parent and child must be in the same solution; child flow trigger = "Manually trigger a flow" with defined input/output schema
  • Do Until patterns: use for polling — set a "termination condition" (status eq 'Completed') and a loop limit to avoid infinite loops in production
  • Adaptive Card for Teams: use "Post an Adaptive Card and wait for a response" action — outputs approval/response JSON; more flexible than the Approvals connector

High-Frequency Interview Soundbites

  • Why SPFx over SPFX Add-ins? — SPFx runs client-side in the page context; no iFrame, same trust as the page, access to SharePoint APIs without cross-origin issues, full React/TypeScript stack.
  • Difference between Application Customizer and Field Customizer? — App Customizer injects UI into named placeholders (Top/Bottom of page). Field Customizer replaces the rendered cell value for a specific list column.
  • How does SPFx get SharePoint context? — Through this.context (BaseWebPartContext / BaseExtensionContext) populated by the SharePoint page runtime. Contains pageContext, spHttpClient, msGraphClientFactory, serviceScope.

  • How do you update a deployed web part without breaking existing instances? — Never change the component GUID in the manifest. Bump the version in package-solution.json. Redeploy. Instances pick up the new JS bundle automatically because they reference by GUID, not bundle filename.
  • What's the list view threshold and how do you work around it? — 5,000 items default. Use indexed columns, filtered views, or server-side pagination with PnP getPaged(). For bulk operations, use batch mode or Power Automate with chunking.
  • How do you pass a config value from property pane to a React component? — Store in web part class properties (this.properties.myProp), pass as React props on render, re-render is triggered by onPropertyPaneFieldChanged which calls this.render().

Q 3.1
What's the difference between Automated, Instant, Scheduled, and Solution-aware flows? Which would you use for a tenant-wide approval routing system?
  • Automated — triggered by an event (list item created, file uploaded).
  • Instant — manually launched by user via button or "Run a flow" pane.
  • Scheduled — recurrence (nightly cleanup).
  • Solution-aware — packaged inside a Dataverse solution. Required for environment promotion (Dev → Test → Prod) and ALM.

For a tenant-wide approval routing system I always build solution-aware, even if it starts small. Without solutions, connection references hard-code per-environment values and there's no clean export path. Painful to retrofit later.

Q 3.2
How do you handle errors in Power Automate? Walk through Try/Catch/Finally pattern.

Use Scope actions configured with RunAfter conditions:

  • Try Scope — happy path
  • Catch Scope — RunAfter = Try has failed, skipped, or timed out
  • Finally Scope — RunAfter = Catch is any (succeeded, failed, skipped) — ensures cleanup logging always runs

Inside Catch, I capture: workflow()?['run']?['name'], result('Try_Scope') (which returns all failed actions), the trigger output, and push to App Insights or a SharePoint "FlowErrors" list. This is the difference between a flow that silently fails and one that gets fixed.

// Result expression for catch logging
filter(result('Try_Scope'), item()?['status'] eq 'Failed')
Q 3.3
What is the SharePoint "Send an HTTP request to SharePoint" action used for, and when is it better than the standard SharePoint connector?

It calls SharePoint REST or CSOM endpoints directly using the flow's connection. Use it when:

  • Standard connector doesn't support what you need (e.g., set field default values, modify list view, break permission inheritance, batch operations)
  • You need to call _api/web/lists('id')/breakRoleInheritance(true)
  • You're hitting connector throttling — direct REST often has different limits
  • You need $batch to combine 50 operations into one request
// Example: assign unique permissions to a folder

Q 3.4
A flow runs 500 times an hour and is hitting SharePoint connector throttling. How do you fix it?
  1. Batch via "Send HTTP request" — replace 50 "Update item" actions with one $batch call.
  2. Reduce trigger frequency — change "When item is created" to "When item is created OR modified" with a status filter, so flows don't fire for irrelevant updates.
  3. Trigger conditions — use trigger conditions to fail fast without consuming an API call: @equals(triggerBody()?['Status']?['Value'], 'Submitted').
  4. Use Dataverse for high-volume state — SharePoint lists were never designed for 100K rows + sub-second writes.
  5. Premium connectors with higher limits — HTTP with AAD, or move computation to Azure Functions called from the flow.
  6. Concurrency control on Apply to Each — set Degree of Parallelism to a value below the connector's per-second limit.
Q 3.5
What is the difference between Concurrency Control on an "Apply to each" and Sequential mode? When does sequential matter?

By default, Apply to each runs items sequentially. Toggling "Concurrency Control" lets you set Degree of Parallelism (1–50).

Stay sequential when:

  • Order matters (e.g., updating a "Position" column 1, 2, 3...)
  • You're calling a downstream system that does not support concurrency (legacy SOAP API)
  • Each iteration depends on the previous iteration's output

Go parallel when: items are independent (sending 200 emails). I default to 20 for SharePoint and 5 for Dynamics — adjust based on observed throttling.

Q 3.6
Explain retry policies on actions. Default behaviour vs custom configuration.

Default = Exponential retry, 4 attempts, intervals of 5s, 10s, 20s, 40s (roughly). You can override per action:

  • None — never retry. Use for non-idempotent operations (sending email, charging payment).
  • Default (exponential) — leave for most idempotent calls.
  • Fixed Interval — useful when downstream system needs a steady cadence.
  • Exponential with custom count + min/max interval — for finicky APIs (e.g., 8 retries, 1 minute base).
Anti-pattern Don't crank retry count up to 10 to "hide" a flaky integration. The flow keeps running but the user experience is a 4-minute delay. Fix the root cause or build a circuit breaker.
Q 3.7
How do you implement secure secrets in Power Automate flows?
  • Azure Key Vault connector — flow pulls secret at runtime, never stored in flow definition.
  • Environment variables (solution-aware) — type "Secret Reference" pointing to Key Vault. The reference moves through Dev/Test/Prod cleanly.
  • Secure Input / Secure Output — toggle on HTTP actions so secrets don't appear in run history. Important for audit / GDPR.
  • Never hard-code in compose actions, variables, or list rows. I've seen API keys in SharePoint lists "for convenience" — that's a fireable offence in regulated industries.
Q 3.8
Difference between Standard and Premium connectors. Cost implications.

Standard connectors (SharePoint, Outlook, Teams, OneDrive, Approvals) are included with Microsoft 365 licenses. Premium connectors (HTTP, SQL, Dataverse, custom connectors, on-premises data gateway, all third-party SaaS like Salesforce, ServiceNow, SAP) require a Power Automate Premium license per user OR a per-flow license.

Cost reality at enterprise scale: a flow that uses an HTTP action is Premium. If 10,000 users are licensed for Power Automate Premium at ~$15/user/month, that's $1.8M/year. The architecture decision "do we use HTTP or build a custom Standard wrapper" can be a million-dollar conversation.

Mitigation: use per-flow plans (flat fee per flow, unlimited users) for tenant-wide automation that's used by everyone — usually much cheaper.

Q 3.9
What is a Custom Connector and when should you build one?

A Custom Connector wraps a REST API (defined via OpenAPI/Swagger or Postman collection) so it appears alongside built-in connectors in flow designer, with proper auth, action discovery, and parameter typing.

Build one when:

  • The API is reused across >3 flows
  • You need consistent auth handling (OAuth2, API key, Basic) without re-configuring per flow
  • You want non-developers to consume the API in low-code style
  • You want to publish to AppSource or share across the org via solutions

Don't build one for a single one-off call — just use HTTP action with environment variables.

Q 3.10
How do you achieve idempotency in a flow that processes SharePoint items?

Flows can fire twice (rare but real — e.g., during platform throttling recovery). Idempotency strategies:

  • Status field — set ProcessingStatus = "InProgress" at start, only process if "NotStarted". Combined with trigger condition: @equals(triggerBody()?['ProcessingStatus']?['Value'], 'NotStarted').
  • Idempotency key — generate guid() at trigger, store with the item, check before downstream call.
  • Conditional update with ETag — when calling SP REST, pass If-Match header. SP returns 412 Precondition Failed if the item changed since you read it. Treat that as "another instance handled it" and exit gracefully.

Q 4.3
What pre-migration assessment do you run before quoting timelines?
  1. Content inventory — total GB, file count, max item per list, max files per folder, file types (especially .aspx, .htm with embedded JS).
  2. Customization scan — SharePoint Modernization Scanner (free, PnP), gives you a CSV of every web part, master page, workflow, web template, info path form.
  3. Workflow scan — count SP Designer 2010/2013 workflows; each is a rebuild task.
  4. Permission map — broken inheritance, security groups, fine-grained item-level permissions (a red flag for large lists).
  5. Branding inventory — master pages, custom page layouts, theme files.
  6. External dependencies — BCS connections, SQL ODBC sources, external lists.

Q 4.4
How do you handle SharePoint 2010/2013 workflows during migration?

Both are deprecated in SPO. The path:

  • Inventory — Get-PnPWorkflowDefinition, classify by complexity.
  • Simple workflows (notify on item change) — convert to Power Automate using the SharePoint connector. Low effort.
  • Approval workflows — convert to Power Automate Approvals action. Recreate routing logic.
  • Complex state-machine workflows — usually a redesign. Sometimes I move them to Logic Apps for the more complex orchestration patterns.
  • Workflow history — if compliance requires retention of historical workflow runs, export to a SQL database before decommissioning. SPO does not import legacy workflow history.
Q 4.5
Authentication differences: Classic SharePoint (NTLM/Kerberos) vs SharePoint Online (Modern auth / AAD). What breaks?

On-prem typically uses Windows Auth (NTLM, Kerberos, or claims-based with ADFS). SPO uses Azure AD with Modern Authentication (OAuth2 / OpenID Connect).

What breaks during/after migration:

  • Hardcoded AD group SIDs in custom code — must remap to AAD groups (different identifier).
  • Trusted login providers — federation with on-prem ADFS must be replaced by Azure AD Connect or Entra B2B.
  • App-only authentication — old SharePoint App Model with client_id/client_secret is deprecated in favour of Azure AD app registrations with certificates.
  • External users — partner users were often added directly to AD; in SPO they need to be guests in AAD via B2B invitation.
  • CSOM/REST scripts using SharePointOnlineCredentials (basic auth) — basic auth is disabled in SPO since 2022. Must migrate to MSAL or PnP PowerShell with cert auth.
Q 4.6
What's the difference between SharePoint REST API on-prem vs SPO? Anything you actually need to rewrite?
  • Auth header — Bearer token (OAuth) replaces NTLM/cookie auth.
  • Form Digest — still required for POSTs. PnP JS handles it; raw fetch must call _api/contextinfo first.
  • Throttling — SPO returns 429 with Retry-After header. On-prem rarely throttled at this layer. Add retry logic everywhere.
  • Some endpoints removed — _vti_bin ASMX web services are gone in SPO. Anything calling /Lists.asmx must move to REST or Graph.
  • Maximum list view threshold — 5000 in both, but in SPO you can't override at farm level. Must use indexed columns + paged queries.
  • Cross-site queries — search APIs largely the same; KQL syntax compatible.
Q 4.7
How do you deal with the 5000 item list view threshold during migration?

The threshold isn't lifted in SPO. Strategies:

  1. Indexed columns — add indexes on filter/sort columns before migration. Without indexes, even small queries blow up on a 100K-item list.
  2. Modern views with row limits — set page size 30; SPO will use indexes automatically.
  3. PnP JS paging — fetch with .top(500).getPaged() rather than loading all items.
  4. Folder structure — folders implicitly partition lists. A library with 500K items in 50 folders queries cheaply if filtered by folder.
  5. Archive split — split into a "Live" list (active items) + "Archive" list (read-only). Surface in UI via SPFx web part that queries both.
  6. Move to Dataverse — if the list is really a database table, get it out of SharePoint.
Q 4.8
A document library has 1.2 million files. How do you migrate it?
  1. Pre-create destination library with correct content types and column structure, and add indexes.
  2. Use SPMT or ShareGate in parallel streams — partition by folder, run 5–10 parallel jobs against different folder paths. Stay under SPO's per-tenant migration API throttle (~3 GB/min).
  3. Delta sync — for the cut-over window, run one final delta pass for files modified since the bulk migration started.
  4. Versions — decide whether to migrate version history (massive size cost) or "latest only" (faster). Compliance dictates this.
  5. Validate — random sample 1% of files: open, check metadata, check permissions, check author/created.
  6. Plan downtime windows — 1.2M files typically takes 5–14 days at ~5–10 files/sec sustained.
Q 4.9
What is "Hybrid SharePoint" and when would you keep it long-term?

Hybrid SharePoint connects on-prem (2013/2016/2019/SE) with SPO via federation: hybrid search (unified results), hybrid OneDrive (redirect to SPO), hybrid taxonomy and content types, hybrid sites following.

Keep hybrid long-term when:

  • Regulatory restriction prohibits some workloads in cloud (banking, defence)
  • Massive content already on-prem with no business case to move (5+ PB archives)
  • Custom server-side code that the business has refused to rewrite

Most enterprises treat hybrid as a transition state, not destination. After 2–3 years, the cost of running both farms exceeds the migration cost.

Q 4.10
How do you migrate InfoPath forms?

InfoPath is dead. Options:

  • Power Apps Customized List Form — quick win for simple forms.
  • SPFx Form Customizer — for forms with cascading dropdowns, complex validation, REST integration.
  • Power Apps Canvas App — when the form is really an app with multiple screens and offline support.
  • Power Apps Model-driven — only if you're moving to Dataverse anyway.

InfoPath XSN files cannot be machine-translated. Plan effort as: 4–8 hours per simple form, 1–3 days per complex form. A typical org has 200–800 InfoPath forms — this is its own programme.

Q 4.11
How do you preserve item-level permissions during migration?

SPMT and ShareGate both preserve unique permissions if the source identity maps to a destination identity. The gotchas:

  • Run Azure AD Connect before migration so on-prem AD identities resolve to AAD users.
  • Map orphaned users to a "Migration Cleanup" group — items with deleted source users will fail otherwise.
  • Lists with thousands of items having unique permissions are slow — consider whether the permission model can be re-architected during migration.
  • Audit the result: Get-PnPListItem -List X | ForEach { Get-PnPListItemPermission ... } on a sample.
Q 4.12
What's your rollback plan if the migration goes wrong at 60% completion?

Real answer: there is no clean rollback once content is in SPO and users have started editing it. Defences:

  • Don't cut DNS until validated — keep on-prem read-only as the system of record until SPO is validated.
  • Phased cutover — migrate per business unit, not per farm. A bad migration of HR doesn't break Finance.
  • SPO backups via M365 Backup or third-party (AvePoint, Veeam) — native SPO retention is not a backup. Get a real backup before cutover.
  • Read-only freeze on the source during cutover window — prevents new edits being orphaned.
  • "Go" / "No-Go" checkpoint after each batch — define quantitative pass criteria (file count match, permission count match, %1 sample passes).

I've never had to fully roll back, but I've stopped a migration at 30% in twice and that's the discipline that saved both projects.

Leadership & Architecture

Team of 5+, code review discipline, production incidents, stakeholder conversations.
Q 6.1
You're leading 5 developers on a migration. Two are senior, two mid, one junior. How do you distribute work?

Skill-based slicing, not hierarchy-based:

  • Senior 1 — architecture, hardest scenarios (form customizers, custom auth, perf-critical web parts), code review lead.
  • Senior 2 — DevOps + CI/CD pipeline + migration tooling automation. Pair with junior on shared services so the junior absorbs patterns.
  • Mid 1 + Mid 2 — split web parts and extensions backlog. One owns extensions track, one owns standard web parts.
  • Junior — pairs with seniors on complex work; owns straightforward web parts + writes Storybook + tests. Visible scope, low blast radius, mentored.

I refuse to put the junior on unsupervised production-critical paths. That's not protectiveness — it's risk management.

Q 6.2
What's your code review philosophy and process?
  • Mandatory pre-merge review — no exceptions, including for me as lead. PRs <400 lines preferred; larger get rejected for splitting.
  • Two-tier review — automated (linting, type check, unit tests, SonarQube, accessibility checks) before human eyes touch it.
  • Reviewer checklist — security (XSS, auth, secrets), performance (avoidable network calls, render hot paths), maintainability (naming, dead code), test coverage.
  • "Strong opinions, loosely held" — I push back on patterns but explain why. Reviewer disagrees? I take it offline; we don't argue in PR comments.
  • Pair programming for the hardest 5% — anything novel or risky is paired, not reviewed.

Anti-pattern I've banned: "LGTM" without comments. If you reviewed 600 lines in 90 seconds, you didn't review them.

Q 6.3
A production SPFx web part is down. CEO is asking questions. Walk through your first 30 minutes.
  1. Minute 0–2: Acknowledge to the business: "We're aware, investigating, ETA in 15 min." Calm beats accuracy.
  2. Minute 2–10: Triage. Check Microsoft Service Health (is it the platform?), App Insights traces (do we see the error?), CDN deployment history (did we ship in the last hour?).
  3. Minute 10–20: Decision: rollback or fix-forward? If we shipped in the last 24h, rollback the App Catalog .sppkg version. Otherwise, isolate.
  4. Minute 20–30: Mitigate. If users can't get past it, deploy a tenant-wide "feature flag off" via tenant property bag — the Application Customizer reads that flag and hides itself.
  5. Throughout: Document timeline in a running incident channel. The post-mortem starts now, not at end of incident.

The CEO's question gets a written update at minute 15 with timeline + ETA + scope. Not a phone call — written, because the rest of the org needs the same info.

Q 6.4
How do you communicate technical risk to non-technical stakeholders?

Three rules:

  • Translate to money and time. Not "the API has a 600 req/min throttle" — "if we push 50K records nightly without batching, the sync takes 14 hours instead of 40 minutes; we'd miss the morning report SLA twice a week."
  • Offer options with trade-offs, not just risks. "We can ship Monday with this caveat, or invest 5 dev-days and ship Friday cleanly. Both are valid; here's what I recommend and why."
  • Be specific about probability. "Likely," "possible," "edge case" — I attach a rough percentage. Stakeholders make better decisions with "30% chance" than "could happen."
Q 6.5
How do you handle a senior developer pushing back on your architectural decision?

Listen first. If the pushback has merit, change the decision and credit them. If it doesn't, explain the constraints they may not see (compliance, licensing, executive direction) and ask if those change their view.

When we still disagree:

  • Disagree and commit — make the call, ask them to support it publicly, revisit in two weeks with data.
  • If they're consistently right and I'm consistently overruling — that's a signal I should reconsider my approach, not double down.

The team I lead must be able to disagree without consequence. If a junior dev fears me, I have failed.

Q 6.6
What's your ALM and CI/CD setup for SPFx?
  • Source control: Azure DevOps Git or GitHub. Branch policy: PR required, 1+ reviewers, build must pass.
  • Pipelines: gulp bundle --ship + gulp package-solution --ship produces the .sppkg.
  • Multi-environment deployment: separate App Catalogs for Dev, UAT, Prod (or separate tenants entirely for high-stakes work).
  • Deployment via CLI for M365: m365 spo app add + m365 spo app deploy in pipeline.
  • Versioning: auto-increment version in package-solution.json from build number.
  • Quality gates: ESLint, Prettier, SonarQube, unit tests via Jest, accessibility via axe-core. Pipeline fails on any.
Q 6.7
How do you onboard a new developer to your SPFx project?
  1. Day 1: Environment setup script (Node via nvm, gulp, tenant access, dev cert). README has it scripted — should take <2 hours.
  2. Day 2: Architecture walkthrough — 1 hour with me on a whiteboard. Folder structure, service layer, deployment pipeline.
  3. Days 3–5: Pair with a senior on a small bug fix end-to-end — PR through deployment.
  4. Week 2: First independent ticket, scoped small. Daily 15-min check-in.
  5. Week 4: Code review responsibilities added.

An onboarding checklist lives in the repo. If a new joiner finds a gap, they update it as part of week 1. Compound improvement.

Q 6.8
Tell me about a time you had to say no to a stakeholder.

Recently a business sponsor wanted real-time push notifications for every list item change to 4,000 users, via SPFx + signalR. The signalR alone wasn't the issue — the social cost of 200K notifications/day to 4K users was. I declined the initial scope and proposed:

  • Per-user subscription model (you opt into specific items/lists)
  • Digest delivery (max one notification per item per user per 15 min)
  • Email fallback for off-hours

That gave the sponsor a usable system at a quarter of the engineering cost. Saying no is rarely "no" — it's "not that, and here's what works."

Common Mistakes & Best Practices

The anti-patterns that sink SPFx projects — and the exact fixes. Read these as a candidate to avoid them; read them as an interviewer to probe whether candidates have lived through them.
Format: each entry is MISTAKE → WHY IT HURTS → FIX, with code where a snippet makes the fix obvious.
M 11.1
Calling spfi() inside every component render instead of once in onInit
Anti-pattern: const sp = spfi().using(SPFx(props.context)); inside the component body — creates a new PnP instance on every re-render, destroys caching, leaks memory.
Fix: initialize once in onInit and pass sp via props or React Context, or use a module-level singleton guarded by a null check.
// WebPart.ts — onInit
private _sp: ReturnType<typeof spfi>;

public async onInit(): Promise<void> {
  await super.onInit();
  this._sp = spfi().using(SPFx(this.context));
}

public render(): void {
  ReactDOM.render(
    <MyComponent sp={this._sp} />,
    this.domElement
  );
}
M 11.2
Forgetting to cancel async operations in useEffect cleanup
Anti-pattern: calling setState after a component has unmounted — causes React "can't perform state update on unmounted component" warnings and memory leaks in long-lived SPFx pages.
Fix: use a cancelled flag or an AbortController.
useEffect(() => {
  let cancelled = false;
  sp.web.lists.getByTitle("Tasks").items()
    .then(items => {
      if (!cancelled) setItems(items);
    });
  return () => { cancelled = true; };
}, []);
M 11.3
Using any types everywhere in TypeScript
Anti-pattern: const item: any = await sp.web.lists... — defeats the compiler, hides bugs until runtime, and signals to reviewers that the dev doesn't trust their own data shapes.
Fix: define interfaces that match your list columns. For PnP responses, use the generic overload: sp.web.lists.getByTitle("X").items<IMyItem[]>().
interface ITask {
  Id: number;
  Title: string;
  AssignedToId: number;
  DueDate: string;
}

const items = await sp.web.lists
  .getByTitle("Tasks")
  .items.select("Id", "Title", "AssignedToId", "DueDate")
  <ITask[]>();
M 11.4
Over-fetching list data — no $select, no $top
Anti-pattern: sp.web.lists.getByTitle("X").items() with no selectors — returns every column including heavy lookup expansion, system fields, and potentially thousands of rows. Kills performance, hits list view threshold.
Fix: always chain .select(), .top(), and .filter(). For large lists use server-side pagination with .getPaged().
const page = await sp.web.lists
  .getByTitle("Projects")
  .items
  .select("Id", "Title", "Status", "Author/Title")
  .expand("Author")
  .filter("Status eq 'Active'")
  .top(100)
  .getPaged<IProject[]>();
const items = page.results;
if (page.hasNext) { const next = await page.getNext(); }
M 11.5
Blocking the render cycle with synchronous-looking awaits at the top level
Anti-pattern: making the render() method async and awaiting data before calling ReactDOM.render — the web part shows nothing until the await resolves, giving users a blank box.
Fix: render immediately with a loading state; fetch inside useEffect or a custom hook and update state when data arrives.
M 11.6
Not calling ReactDOM.unmountComponentAtNode in onDispose
Anti-pattern: omitting onDispose — the React component tree keeps living in memory after the web part is removed from the page. In SharePoint pages with dynamic web part manipulation (e.g., News), this causes progressive memory leaks.
Fix: always implement onDispose.
public onDispose(): void {
  ReactDOM.unmountComponentAtNode(this.domElement);
  super.onDispose();
}
M 11.7
Using innerHTML / dangerouslySetInnerHTML to render user-supplied content
Anti-pattern: injecting list column data directly into HTML without sanitization — classic XSS vector in SharePoint. A malicious list editor writes <script>alert(1)</script> into a column and it runs for every viewer.
Fix: never use raw innerHTML. If rich text is genuinely required, use DOMPurify.sanitize() before passing to dangerouslySetInnerHTML. For the PnP Rich Text control, it handles this internally.
M 11.8
Hardcoding site URLs, list GUIDs, or tenant names in component code
Anti-pattern: const url = "https://contoso.sharepoint.com/sites/hr" hardcoded inside a component — breaks in every other environment (dev, UAT, another tenant's deployment).
Fix: Use this.context.pageContext.web.absoluteUrl for current-site relative URLs. For cross-site references, expose them as configurable web part properties in the property pane.
// In property pane:
{
  groupName: "Configuration",
  groupFields: [
    PropertyPaneTextField("targetSiteUrl", {
      label: "Target Site URL"
    })
  ]
}
M 11.9
Deploying a web part that requires Graph permissions without admin consent
Anti-pattern: declaring webApiPermissionRequests in package-solution.json and deploying, assuming it just works — the permission request is created in "Pending" state; calls fail with 401 until a Global Admin or SharePoint Admin approves it in the API Access page.
Fix: coordinate with your tenant admin at deploy time. Include approval instructions in your deployment runbook. In automated pipelines, use m365 spo serviceprincipal grant add with admin credentials after package deploy.
m365 spo serviceprincipal grant add \
  --resource "Microsoft Graph" \
  --scope "User.Read.All"
M 11.10
Using global CSS class names instead of CSS Modules in SPFx
Anti-pattern: writing .myButton { color: red; } in a .scss file imported without CSS Modules — the class leaks to the entire SharePoint page, breaking styles in every other web part and SharePoint chrome.
Fix: use .module.scss files and reference via the generated object: styles.myButton. SPFx scopes these to unique class names at build time.
// ✓ CORRECT — MyWebPart.module.scss
.myButton {
  background: var(--colorBrandBackground);
  color: white;
}

// ✓ CORRECT — Component usage
<button className={styles.myButton}>Click me</button>
M 11.11
Migration pitfall: translating AngularJS $watch as a useEffect with no dependency array
Anti-pattern: useEffect(() => { fetchData(); }) — no dependency array means it runs after every render, hammering the SharePoint REST API and re-rendering infinitely if the fetch sets state.
Fix: provide an explicit dependency array. If you truly want "run when X changes", list X. If you want "run once on mount", use [].
// AngularJS: $scope.$watch('filterValue', fetchData)
// React equivalent:
useEffect(() => {
  fetchData(filterValue);
}, [filterValue]); // ← explicit dep, not missing []
M 11.12
Migration pitfall: porting AngularJS two-way binding to uncontrolled React inputs
Anti-pattern: using defaultValue and reading via ref.current.value — uncontrolled inputs drift out of sync with state in complex forms, making validation and conditional logic unreliable.
Fix: use controlled inputs: value={state} onChange={e => setState(e.target.value)}. Fluent UI's TextField is controlled by default.
M 11.13
Not handling throttling (HTTP 429) in Power Automate flows that call SharePoint at scale
Anti-pattern: looping over 500 list items and calling "Update item" inside the loop with default concurrency (1) and no retry policy — flow runs for hours and silently fails on 429 responses after the SharePoint throttle kicks in.
Fix: (1) Wrap the loop body in a Try/Catch scope; (2) set retry policy to "Exponential interval" (count: 5, interval: PT5S); (3) use batching — collect updates and call SharePoint $batch endpoint or the "Send an HTTP request to SharePoint" action with batched JSON.
M 11.14
Deploying to App Catalog without skipFeatureDeployment: true and wondering why the web part isn't available site-wide
Anti-pattern: uploading to tenant App Catalog, setting "Make this solution available to all sites" — but if skipFeatureDeployment is false in package-solution.json, site owners still have to manually "Add an app" per site.
Fix: set "skipFeatureDeployment": true in package-solution.json for solutions that should be immediately available everywhere after tenant-wide deployment.
{
  "solution": {
    "name": "my-solution",
    "id": "...",
    "version": "1.0.0.0",
    "skipFeatureDeployment": true
  }
}
M 11.15
Forgetting that onSave() / onClose() must be called in a Form Customizer
Anti-pattern: saving the list item via PnP JS but not calling this.context.formCustomizerContext.onSave() — the panel stays open, the list view doesn't refresh, and the user doesn't know if the save succeeded.
Fix: always call onSave() after a successful item add/update, and onClose() after a deliberate cancel. These are the lifecycle hooks that tell SPFx to close the panel and signal the page.
M 11.16
Using sp.web.currentUser.get() for every component that needs the current user
Anti-pattern: each component independently calls sp.web.currentUser.get() — results in N parallel calls for N components on the same page.
Fix: read the current user once from this.context.pageContext.user (already available, no HTTP call) or centralise the fetch in a React Context provider at the web part root and pass the result down.
// Free — already populated by SPFx page context
const { displayName, email, loginName } =
  context.pageContext.user;
M 11.17
Including large third-party libraries in an SPFx bundle without code-splitting
Anti-pattern: importing Lodash, Moment.js, or a full charting library — they are bundled into the web part JS, inflating a .js from 150 KB to 1.5 MB and breaking SharePoint's loading performance budget.
Fix options: (1) Use SharePoint's already-loaded packages from @microsoft/sp-core-library; (2) use tree-shakeable imports (import debounce from 'lodash/debounce'); (3) load heavy libs as externals via config/config.json with a CDN URL; (4) replace Moment with native Intl.DateTimeFormat or date-fns.
M 11.18
Not version-bumping package-solution.json before redeploying
Anti-pattern: rebuilding and re-uploading the .sppkg without bumping the version field — SharePoint App Catalog may serve the cached old version to sites that already added the app, causing production bugs from stale code.
Fix: increment the version on every production deploy (1.0.0.0 → 1.0.0.1). Automate this in your CI pipeline using a script or the --increment flag in custom Gulp tasks.
M 11.19
Registering Application Customizer in the wrong scope during testing
Anti-pattern: adding the custom action at Site scope thinking it will appear on all subsites — in SPFx, Application Customizers registered at Site (site collection root) scope only apply to the root web, not subsites. Tenant scope applies everywhere.
Fix: for tenant-wide header/footer, use --scope Tenant via CLI for M365 or SPFx tenant-wide deployment with skipFeatureDeployment: true.
M 11.20
Running gulp serve against the wrong serve.json environment
Anti-pattern: editing config/serve.json to point at the dev tenant and committing that change — teammates pull the repo and their serve session starts pointing to your tenant. Causes confusing 403 errors and data mutations in the wrong environment.
Fix: add config/serve.json to .gitignore. Commit a serve.json.sample with placeholder values. Each developer fills in their own local copy.

Fluent UI & SASS Styling

The JD explicitly calls out Fluent UI, PnP Controls, and SASS. These 13 questions cover component selection, theming, v8 vs v9, CSS Modules, SharePoint theme tokens, and responsive patterns — the layer most architects underestimate.
FU 14.1
What is Fluent UI and why is it the preferred choice for SPFx web parts over Bootstrap or custom CSS?

Fluent UI (formerly Office UI Fabric) is Microsoft's open-source design system built for the Microsoft 365 ecosystem. In SPFx it is the right choice because:

  • SharePoint-native look and feel: Components match the SharePoint Online chrome, typography, and color system out of the box — no visual jarring.
  • Theme inheritance: Fluent UI components automatically pick up the SharePoint site's theme (colours set by the site owner) via the SharePoint theme tokens. No custom CSS needed to respect the branding.
  • Accessibility by default: All core components implement ARIA roles, keyboard navigation, and focus management to WCAG 2.1 AA. Building this yourself is months of work.
  • Already in the bundle: React and Fluent UI are loaded by SharePoint's page already; they don't inflate your web part's bundle size (if you declare them as externals).
  • RTL support: Automatic right-to-left layout via the dir attribute — critical for Arabic/Hebrew tenants.
FU 14.2
Fluent UI v8 vs v9 — what's the difference and why can't we use v9 in SPFx yet?
  • Fluent UI v8 (@fluentui/react v8): Class-based internals, uses the older mergeStyles CSS-in-JS engine, ships with React 16/17, stateful components like Dropdown, DetailsList, Dialog. This is what SPFx ships with.
  • Fluent UI v9 (@fluentui/react-components): Complete rewrite, requires React 18, uses Griffel (CSS-in-JS at build time), fully headless/composable, much smaller bundle per component. Not yet compatible with SPFx because SPFx 1.18 ships with React 17.
Current state (SPFx 1.18, 2024): Use @fluentui/react v8. Do not install v9 — it will pull React 18 into your bundle and create version conflicts. Microsoft is working on SPFx support for v9 with a future SPFx release.
Interview signal: A candidate who knows this distinction has actually tried to use v9 and hit the wall — that's experience, not just reading the docs.
FU 14.3
How do you apply a custom theme in SPFx using Fluent UI's ThemeProvider and createTheme?
import { ThemeProvider, createTheme } from "@fluentui/react";

const myTheme = createTheme({
  palette: {
    themePrimary: "#0b5fff",
    themeDarker: "#083ea6",
    neutralPrimary: "#14131f",
  },
  defaultFontStyle: { fontFamily: "Inter, sans-serif" },
});

// Wrap the web part's root component
export const MyWebPart: React.FC<IProps> = (props) => (
  <ThemeProvider theme={myTheme}>
    <MainContent {...props} />
  </ThemeProvider>
);

Alternatively, use useTheme() hook inside child components to read the active theme, or loadTheme() globally. For most SPFx solutions, simply not overriding the theme lets SharePoint's site theme flow through automatically via CSS variables.

FU 14.4
When do you use Panel vs Dialog vs Modal in Fluent UI?
  • Panel: Slides in from the side (right by default). Use for forms, item details, multi-step wizards — anything requiring significant real estate without leaving the page context. The Form Customizer effectively replaces the built-in panel.
  • Dialog: Centered overlay for focused decisions — "Are you sure you want to delete?", "Confirm submission". Keep dialogs short; one or two actions maximum.
  • Modal: Base overlay primitive without Dialog's opinionated structure. Use when you need full control over the overlay content. Dialog wraps Modal internally.
// Panel — best for item edit forms in list web parts
<Panel
  isOpen={isEditing}
  type={PanelType.medium}
  headerText="Edit Task"
  onDismiss={() => setIsEditing(false)}
  isFooterAtBottom={true}
  onRenderFooterContent={() => (
    <><PrimaryButton onClick={handleSave}>Save</PrimaryButton>
       <DefaultButton onClick={() => setIsEditing(false)}>Cancel</DefaultButton></>
  )}
>

FU 14.6
How does Stack and StackItem work for layout and when do you prefer it over CSS Flexbox?

Stack is Fluent UI's layout primitive — a div with pre-wired flexbox semantics and spacing tokens. Use it for predictable, accessible spacing inside web part forms and card layouts.

FU 14.7
What is mergeStyles from @fluentui/merge-styles and when would you use it instead of CSS Modules?

mergeStyles is Fluent UI's CSS-in-JS engine — it generates scoped class names at runtime from a style object, similar to emotion or styled-components. It is the approach Fluent UI itself uses internally.

import { mergeStyles } from "@fluentui/merge-styles";

  background: "white",
  borderRadius: 4,
  padding: "16px",
  boxShadow: "0 2px 8px rgba(0,0,0,.12)",
  "&:hover": { boxShadow: "0 4px 16px rgba(0,0,0,.18)" },
});
When to choose: Use mergeStyles when styles depend on props or runtime theme values (dynamic styling). Use CSS Modules (.module.scss) for static, predictable styles — it gives you SASS features (variables, mixins, nesting) and the styles are extracted at build time, not injected at runtime.

How do you consume SharePoint's site theme colours in SASS within SPFx?

SharePoint injects CSS custom properties (CSS variables) for the current site theme into the page at runtime. The Fluent UI / SPFx toolchain also exposes them via [theme:…] token syntax in SASS files, which gets replaced at build time.

/* Method 1 — CSS custom properties (preferred, runtime) */
.myButton {
  background-color: var(--spx-color-themePrimary, #0b5fff);
  color: var(--spx-color-white, #fff);
}

/* Method 2 — SPFx theme tokens (compile-time replacement) */
.myButton {
  background-color: "[theme:themePrimary, default:#0b5fff]";
  color: "[theme:white, default:#fff]";
}
Method 1 (CSS vars) reacts to theme changes at runtime without a page reload. Method 2 is baked in at build. For the best experience in multi-tenant or white-label deployments, always use Method 1 so the web part automatically adapts to any site's colour scheme.

Part B — Classic SharePoint Foundations

CS 15.8
What are master pages and page layouts in Classic SharePoint, and what replaces them in Modern?
  • Master page: A SharePoint-specific ASP.NET master page that defines the outer chrome of every page — navigation, top bar, footer, search. Stored in the Master Page Gallery. Custom master pages were the primary branding mechanism in Classic.
  • Page layout: An ASP.NET content page that defines the structure of a Publishing page — the zones and field controls visible to content authors. Paired with a content type to define available metadata fields.
  • Modern equivalent: Master pages and page layouts don't exist in Modern SharePoint. Branding is handled via the SharePoint theme (colour palette, fonts) configured at site level and applied automatically. Custom chrome is added via Application Customizer Extensions (top/bottom placeholders). Full-page layouts are replaced by Section layouts on the modern page canvas (one column, two column, etc.) with web parts filling the zones.
CS 15.9
Explain farm solutions vs sandboxed solutions vs SharePoint Add-ins vs SPFx — the evolution.
  • Farm solutions (.wsp, full trust): Run in the SharePoint server process with full CLR trust. Could do anything — custom HTTP modules, timer jobs, event receivers, custom web services. Require farm admin deployment. SharePoint Online: not supported.
  • Sandboxed solutions: Run in an isolated process with limited API surface. Attempted to allow site collection–level deployment without farm admin. Deprecated in 2014 — the security model was complex and the API was too limited to be useful.
  • SharePoint Add-ins (Apps): Ran either in SharePoint (SharePoint-hosted, client-side in an iframe) or on a remote server (Provider-hosted). Isolated from the host site via iFrame. OAuth for permissions. Successor to farm solutions for SharePoint Online. The iFrame isolation hurt UX and performance — widely disliked.
  • SPFx: Client-side, runs in the SharePoint page context (no iFrame), full React/TypeScript, same security as the hosting user, peer-reviewed and deployed via App Catalog. The current and future-proof model.
CS 15.10
What is the SharePoint Feature Framework and what kinds of features exist?

A Feature is a packaged unit of SharePoint functionality that can be activated or deactivated at a specific scope. Scopes from widest to narrowest: Farm → Web Application → Site (site collection) → Web (subsite).

Common feature uses: deploying content types, site columns, list templates, master pages, custom actions, workflows. Each feature can have Feature Receivers (event handlers for Activating, Deactivated, Installed, etc.) that run arbitrary code.

In migration context: When migrating Classic to Modern, you audit which custom features exist, what they deploy, and find the Modern equivalent for each capability (SPFx extensions replace Custom Actions; content types persist; list templates can be recreated in Modern).

CS 15.11
What are Content Types and Site Columns in SharePoint, and how do they survive migration to SharePoint Online?

Site Columns: Reusable column definitions defined at site collection level (or higher via Content Type Hub). Any list in the site can add a site column rather than recreating the same column definition with the same business rules.

Content Types: A named collection of site columns plus metadata (description, document template, workflow, event receivers). A list or library can have multiple content types, allowing different item shapes in the same container.

Migration: Site columns and content types migrate well to SharePoint Online — SPMT and ShareGate both preserve them. The gotcha is columns bound to Managed Metadata fields (taxonomy) — these require the same term set GUIDs to exist in the destination tenant's term store before migration. Always migrate the term store before the content.

CS 15.12
What were Custom Actions in Classic SharePoint and what is their SPFx equivalent?
  • ScriptLink custom action: Injected arbitrary JavaScript into every page on a site via a script tag. The primary mechanism for AngularJS-based customizations in Classic SharePoint — a single <script src="app.js"> injected globally. Modern equivalent: Application Customizer.
  • ECB (Edit Control Block) menu items: Added custom items to the right-click context menu on list items. Modern equivalent: ListView Command Set (context menu commands).
  • Ribbon custom actions: Added buttons to the SharePoint ribbon. Modern equivalent: ListView Command Set (toolbar commands).
  • Field rendering override: Used JSLink to override how a column rendered in a list view. Modern equivalent: Field Customizer Extension.
CS 15.13
What is the difference between CSOM, JSOM, and the REST API? What should you know about each for migration?
  • CSOM (Client-Side Object Model): .NET library (Microsoft.SharePoint.Client) for server-side or thick-client SharePoint access. Used heavily in migration tools, timer jobs, and PowerShell scripts. The underlying transport is SOAP-based XML over HTTP. Still fully supported in SharePoint Online for server-side tooling.
  • JSOM (JavaScript Object Model): The JavaScript version of CSOM, loaded as SP.js via SP.SOD (Script on Demand) on Classic SharePoint pages. Used in SharePoint-hosted Add-ins and inline JavaScript customizations. Not used in SPFx — it's a Classic-era API.
  • REST API: OData-based HTTP API, available since SharePoint 2013. Used by SPFx via SPHttpClient, PnP JS, and Power Automate. Future-proof and fully supported in SharePoint Online.

Migration relevance: Classic AngularJS code typically used JSOM or direct REST calls via $http. Migrating to SPFx means replacing JSOM calls with PnP JS v3 (which wraps REST) and service abstractions that the Angular $http service once provided.

A11Y 16.1
What are the WCAG 2.2 conformance levels and which level do Microsoft's own products target?
  • Level A: The minimum. Covers the most basic accessibility requirements — page has a text alternative for non-text content, keyboard navigation is possible, no content causes seizures.
  • Level AA: The legal standard in most jurisdictions (US ADA, EU European Accessibility Act). Adds colour contrast ratios (4.5:1 for normal text, 3:1 for large text), focus indicators, error suggestions in forms, visible labels.
  • Level AAA: The highest level, impractical to achieve site-wide. Includes requirements like sign language interpretation and 7:1 contrast ratios for enhanced accessibility.
Microsoft targets WCAG 2.2 Level AA across SharePoint, Teams, and Fluent UI components. Your SPFx web parts should meet the same bar. WCAG 2.2 adds four new criteria to 2.1 — notably Focus Appearance (2.4.11) and Dragging Movements (2.5.7).

Part B — Performance

PERF 16.6
When does React.memo actually help and when does it make things worse?

React.memo wraps a functional component to skip re-renders if props haven't changed (shallow equality). It helps when: the parent re-renders frequently but the child's output is stable; the child is expensive to render (large lists, charts).

It hurts when: props include objects or arrays created inline — <Child items={[...]} /> creates a new array reference every render, so the memo comparison always fails and you pay the extra comparison cost for nothing. Fix: memoize the prop value with useMemo.

// ✗ Memo is wasted — new array reference every render
const TaskRow = React.memo(({ item }) => <Row item={item} />);
return <TaskRow columns={["Id", "Title"]} item={task} />;

// ✓ Stable reference — memo comparison works
const cols = useMemo(() => ["Id", "Title"], []);
return <TaskRow columns={cols} item={task} />;

Part C — Security

SEC 16.11
How can XSS occur in an SPFx web part and how do you prevent it?

Attack vector: A SharePoint list item contains a malicious script in a text column (e.g., <script>document.cookie...</script>). Your web part reads the column and renders it unsafely.

Prevention layers:

  • React's default escaping: JSX expressions {item.Title} are automatically HTML-escaped — React converts < to &lt;. This protects against XSS in text content by default.
  • Never use dangerouslySetInnerHTML with untrusted content. If you must render HTML (rich text fields), sanitize with DOMPurify.sanitize(html) first.
  • Avoid innerHTML — use React's virtual DOM render path instead.
  • Validate server-side: SharePoint Online's CSP headers and field-level HTML filtering provide a second layer of defense, but never rely solely on the platform — sanitize in your component too.
import DOMPurify from "dompurify";

// Only safe way to render HTML from SharePoint rich text fields
<div dangerouslySetInnerHTML={{
  __html: DOMPurify.sanitize(item.RichContent, {
    ALLOWED_TAGS: ["b", "i", "em", "strong", "a", "ul", "li"],
    ALLOWED_ATTR: ["href", "target"]
  })
}} />
SEC 16.12
Are SPFx web parts vulnerable to CSRF and why?

SPFx web parts run in the same origin as SharePoint and use SPHttpClient, which automatically includes the SharePoint form digest (request digest) in all POST/PATCH/DELETE requests as the X-RequestDigest header. This digest is a CSRF token — it is unique per session, tied to the authenticated user, and expires. Without the correct digest, the server rejects the mutation.

In SPFx 1.14+ this digest management is handled transparently by the framework — you don't need to call _api/contextinfo manually. PnP JS also handles this automatically.

Only a risk if: You make raw fetch() calls to SharePoint endpoints without the digest header. Always use SPHttpClient or PnP JS for SharePoint mutations — they handle CSRF protection for you.
SEC 16.13
Explain the least-privilege principle for webApiPermissionRequests in SPFx.

When your SPFx solution calls Microsoft Graph or a custom Azure AD–protected API, you declare the required OAuth scopes in package-solution.json. These permissions are granted at the tenant level to the SharePoint Online Client Extensibility Web Application Principal — meaning all SPFx solutions in the tenant share the same permission grants.

Least-privilege rules:

  • Request only the scopes you actually use — User.Read instead of User.ReadWrite.All.
  • Prefer delegated permissions (act as the user) over application permissions (act as a service principal with no user context).
  • Document every scope in the solution's README so the admin who approves them understands the blast radius.
  • When removing a permission from a new version, document it — revoking a shared scope might break another solution that also uses it.
Never request: User.ReadWrite.All, Sites.FullControl.All, Directory.ReadWrite.All unless you can justify each one. Over-permissioned SPFx solutions become audit findings and can be rejected at the tenant admin review stage.
SEC 16.14
What is Content Security Policy (CSP) in the context of SharePoint Online and SPFx?

SharePoint Online sends a Content-Security-Policy HTTP header that restricts which origins can load scripts, styles, fonts, and other resources. This is a defence-in-depth measure — even if an attacker injects a script tag, the browser will block it if the source isn't in the CSP allowlist.

SPFx implications:

  • SPFx bundles hosted on the tenant's CDN (SharePoint App Catalog) are on an allowed origin by default.
  • Loading scripts from arbitrary external CDNs (e.g., unpkg.com) in your web part may be blocked by CSP. Use the config/config.json externals CDN path feature which includes the CDN in the solution's trusted sources.
  • Inline scripts and eval() are blocked by SharePoint's CSP. This means libraries that use eval() at runtime will fail — test early.
  • Custom fonts loaded from Google Fonts require a CSP update (SP admin portal → "Allow scripts" settings) in some tenants.
SEC 16.15
How do you securely store and access configuration secrets (API keys, connection strings) in an SPFx solution?
Never do: Hardcode API keys in TypeScript source code or web part properties — web part property values are stored in SharePoint page JSON and readable by anyone with page view access.

Correct approaches (in order of preference):

  • Azure Key Vault + Azure Function proxy: The web part calls an Azure Function (authenticated via AadHttpClient). The Function reads the secret from Key Vault using its managed identity. The web part never sees the secret.
  • Azure AD app + OAuth delegated flow: For API keys that are "per user", use AadHttpClient — the token is obtained silently via the user's existing AAD session. The key lives in the AAD app registration, not in SPFx.
  • SharePoint property bag (site property): Acceptable for non-sensitive configuration (list names, feature flags) that admins set. Not appropriate for secrets — property bags are readable by users with API access.
  • Web part properties: Only for non-sensitive UI configuration (display labels, count limits). Never for keys.

Q 8.A.2
What is the difference between gulp serve and gulp bundle --ship + gulp package-solution --ship?
  • gulp serve — starts a local HTTPS dev server at localhost:4321, serves un-minified JS, hot-reloads on save. Never deploy this output.
  • gulp bundle --ship — produces minified, tree-shaken output in ./temp/deploy ready for CDN upload. The --ship flag switches webpack to production mode.

Q 8.A.3
What is the SPFx context object and what can you get from it?

this.context inside a web part gives access to the entire SharePoint page context at runtime. Key properties:

  • context.pageContext.web.absoluteUrl — current site URL
  • context.pageContext.user.displayName — current user
  • context.pageContext.list.id — list GUID (only when hosted on a list/library page)
  • context.spHttpClient / context.httpClient — authenticated HTTP clients
  • context.msGraphClientFactory — Graph client factory
  • context.aadHttpClientFactory — AAD-secured custom API client factory
  • context.propertyPane — programmatic property pane control
  • context.serviceScope — DI container for shared services

Pass this.context as a prop to your root React component — don't try to import it globally.

Q 8.A.4
What happens when you upload a new version of an .sppkg to the App Catalog?

SharePoint compares the solution version in package-solution.json. If the version incremented, the new .sppkg is accepted and deployed. If you upload the same version, SharePoint may reject the upload or overwrite silently depending on the "Upgrade" vs "Add" action.

CDN propagation is separate — the bundle JS on the CDN can take up to 24h to propagate globally via Office 365 CDN. Version-hashing the bundle filename (webpack's contenthash) is the clean fix for cache busting.

Interview tip: Mention that the App Catalog has no built-in rollback. If you need to revert, you must re-upload the previous .sppkg — which means you should archive every shipped package version in your release pipeline artifacts.

Q 8.A.5
What is ServiceScope in SPFx and why should you use it?

ServiceScope is SPFx's built-in dependency injection container. It allows multiple web parts on the same page to share a single instance of a service (e.g., one PnP context, one Graph client, one caching layer) rather than each constructing their own.

// Register a service
export class ProjectService {
  public static serviceKey = ServiceKey.create<ProjectService>('contoso:ProjectService', ProjectService);
  constructor(serviceScope: ServiceScope) {
    serviceScope.whenFinished(() => { /* init */ });
  }
}

// Consume in web part onInit
const svc = this.context.serviceScope.consume(ProjectService.serviceKey);

Without ServiceScope, 3 web parts on a dashboard all create their own PnP instances and fire 3 identical REST calls for the same list. With ServiceScope, one shared instance fires once.

Q 8.A.6
What is the difference between Local Workbench and Hosted Workbench? Which is still useful?

Local Workbench (localhost:4321/temp/workbench.html) — runs entirely in browser with no SharePoint connection. Only useful for pure React/UI work with mocked data. Microsoft removed it in SPFx 1.13+ because the SharePoint context was always null and the mock was misleading.

Hosted Workbench (https://<tenant>.sharepoint.com/_layouts/15/workbench.aspx) — real SharePoint context, real auth, real APIs. This is the one you use. Still runs your local dev-server bundle via the debug URL param.

In practice I skip workbench for extensions entirely and debug directly on a real modern page with the debug manifest URL appended.

Q 8.B.2
When does useEffect run, and what is the cleanup function for?
  • useEffect(() => {}, []) — runs once after first render (componentDidMount equivalent). Use for initial data fetch, event subscriptions.
  • useEffect(() => {}, [dep]) — runs after first render AND every time dep changes.
  • useEffect(() => {}) — runs after every render. Almost never what you want.

The cleanup function (the return value of useEffect) runs before the component unmounts and before the effect re-runs. Critical in SPFx because onDispose() unmounts the component — without cleanup you'll have:

  • Lingering setInterval timers continuing after the web part is removed from the page
  • SignalR/WebSocket connections not closed
  • React state updates on unmounted components (the "Can't perform state update on unmounted component" warning)
useEffect(() => {
  const id = setInterval(fetchData, 30000);
  return () => clearInterval(id); // cleanup
}, []);
Q 8.B.3
What is useMemo vs useCallback, and when do you use each in SPFx?
  • useMemo — memoizes a computed value. Re-computes only when dependencies change. Use for expensive filtering/sorting of list data so it doesn't re-run on every keystroke.
  • useCallback — memoizes a function reference. Use when passing a callback as prop to a child component wrapped in React.memo — prevents the child re-rendering because "the function is new every render."
// useMemo: filter 5000 list items on search
const filtered = useMemo(
  () => items.filter(i => i.Title.includes(query)),
  [items, query]
);

// useCallback: stable ref for DetailsList onRenderRow
const onRenderRow = useCallback((props) => <CustomRow {...props} />, []);

Don't over-use: memoization has a cost (comparison on every render). Only apply when profiling shows a real issue.

Q 8.B.4
How do you handle loading and error states cleanly in a data-fetching hook for SPFx?
interface AsyncState<T> {
  data: T | null;
  loading: boolean;
  error: string | null;
}

export function useListItems(listName: string) {
  const [state, setState] = useState<AsyncState<IListItem[]>>({ data: null, loading: true, error: null });
  const sp = useSpContext();

  useEffect(() => {
    let cancelled = false;
    setState(s => ({ ...s, loading: true, error: null }));
    spfi().using(SPFx(sp)).web.lists.getByTitle(listName).items()
      .then(data => { if (!cancelled) setState({ data, loading: false, error: null }); })
      .catch(err => { if (!cancelled) setState({ data: null, loading: false, error: err.message }); });
    return () => { cancelled = true; }; // prevent setState after unmount
  }, [listName]);

  return state;
}

The cancelled flag is the lightweight alternative to AbortController for PnP calls.

Q 8.B.5
What is React.memo and when should you apply it to SPFx list-row components?

React.memo is a HOC that does a shallow prop comparison before re-rendering. Wrap a component that is rendered many times (e.g., a list row) and whose props change rarely.

const ProjectRow = React.memo(({ item, onSelect }: IProjectRowProps) => (
  <div onClick={() => onSelect(item.Id)}>{item.Title}</div>
));

If a 500-row list re-renders because a search box state changed at the parent level, without memo all 500 rows re-render. With memo, only rows whose props actually changed re-render. Combine with useCallback on the onSelect handler — otherwise the callback is a new reference every render and memo is defeated.

Q 8.B.6
How do you handle property pane values changing and re-rendering only the affected part of the React tree?

SPFx calls render() on every property pane change. If your root component rebuilds the entire tree, this causes visible flicker and excess API calls.

Pattern: pass only the specific property as a prop; wrap expensive sub-trees in React.memo; move data fetching into a hook that only triggers on the specific dep it needs:

// Web part render() — always called
ReactDOM.render(
  <App
    listTitle={this.properties.listTitle}  // only prop that drives data
    themeColor={this.properties.themeColor}  // only drives style
  />, this.domElement
);

// App — DataPanel only re-fetches when listTitle changes
const DataPanel = React.memo(({ listTitle }) => {
  const { data } = useListItems(listTitle);
  return <ItemList items={data} />;
});

Q 8.C.1
What changed in PnP JS v3 (the @pnp/sp package) that breaks PnP v2 code?

Three breaking changes:

  • No global sp singleton. You now create a factory per request context using spfi().using(SPFx(this.context)). V2 code that did sp.setup(this.context) globally must be rewritten.
  • Invokable queryables. Calling sp.web.lists() as a function (not .get()) is now the pattern. .get() still works but is less idiomatic.
  • Behaviours replace middleware. Features like caching, logging, and retry are composed via .using(Caching(), Logging()) rather than configured globally.
// V2 (old)
sp.setup({ spfxContext: this.context });
const items = await sp.web.lists.getByTitle("Tasks").items.get();

// V3 (current)
const sp = spfi().using(SPFx(this.context));
const items = await sp.web.lists.getByTitle("Tasks").items();
Q 8.C.2
How do you perform a batch of SharePoint REST calls with PnP JS to reduce round trips?
const sp = spfi().using(SPFx(this.context));

// Create a batch
const [batchedSP, execute] = sp.batched();

let webTitle: string;
let listCount: number;

// Queue requests — they don't fire yet
batchedSP.web().then(w => webTitle = w.Title);
batchedSP.web.lists().then(l => listCount = l.length);

// Fire all queued requests as one HTTP call
await execute();
console.log(webTitle, listCount);

One HTTP $batch request replaces 2 separate REST calls. At scale (loading 10 different list item sets for a dashboard), batching can reduce page-load API calls from 10 to 2-3.

Q 8.C.3
How do you add caching to PnP JS calls and what are the gotchas?
import { Caching } from "@pnp/queryable";

const sp = spfi().using(SPFx(this.context), Caching({ store: "session" }));

// This call will be cached in sessionStorage for 5 min (default)
const items = await sp.web.lists.getByTitle("Projects").items.select("Title,Status")();

Gotchas:

  • Caching caches the entire query response keyed on URL. A query with ?$select=Title and one with ?$select=Title,Status are different keys — they won't share cache.
  • Mutations (POST/PATCH/DELETE) are never cached, but they also don't automatically invalidate related GET caches. You must call CacheKey.deleteCache("key") after a write, or use a React Query invalidation pattern.
  • "session" store = sessionStorage (tab-lifetime). "local" = localStorage (persistent). Use session for list data; never use local for user-specific sensitive data.
Q 8.C.4
How do you handle pagination for a list with 50,000 items using PnP JS?

  let page = await sp.web.lists.getByTitle("BigList").items
    .select("ID,Title").top(500).getPaged();

  let allItems = [...page.results];

  while (page.hasNext) {
    page = await page.getNext();
    allItems = [...allItems, ...page.results];

  return allItems;
}

  • EditFormClientSideComponentId

// Using PnP PowerShell to associate

Q 8.G.1

Q 8.G.2
Explain SharePoint's permission inheritance model and how it relates to SPFx.

By default permissions inherit: Tenant → Site Collection → Site → List → Item. When you break inheritance at any level, that object gets its own permission set.

SPFx relevance:

  • Your web part's API calls run as the current user — if the user lacks permissions to a list, PnP calls return 403. Always handle 403 gracefully, not as a generic error.
  • App-Only context (e.g., elevated operations) requires a separate Azure AD app registration — you can't elevate permissions inside SPFx client-side code. Route elevated operations through an Azure Function.
  • The context.pageContext.list.permissions object tells you what the current user can do on the list: hasPermission(SPPermission.editListItems). Use this to conditionally show edit/delete buttons.
Q 8.G.3
What is the SharePoint Search API and when would you use it in an SPFx web part over a list query?

SharePoint Search uses a pre-built index. Queries are fast regardless of list size and can span multiple site collections.

Use Search when:

  • You need cross-site results (e.g., "all Projects across all team sites in the Hub")
  • Full-text search on document content (not possible with list REST API)
  • Ranked/relevance results
  • Large content sets where REST list queries would hit the 5K threshold

Use list REST/PnP when:

  • You need real-time data (Search index has a crawl delay — typically 5–15 min for modified content)
  • You need to write data (Search is read-only)
  • You need all columns including custom managed metadata fields exactly as stored
Q 8.G.4
What are Content Types in SharePoint and how do you work with them in SPFx?

Content types define the schema (columns + form associations) for a SharePoint item or document. They allow multiple lists/libraries across a site collection to share the same structure.

In SPFx:

  • Form Customizer is associated at the content type level — not the list level. One content type = one form experience, regardless of which list uses it.
  • When creating items, pass ContentTypeId to create an item of a specific type in a list that supports multiple content types.
  • Query content type fields via sp.web.lists.getByTitle("X").contentTypes() to dynamically build forms.

Q 1.1
Walk me through the SPFx web part lifecycle. Which hooks have you actually used in production, and which one have you regretted using?

The core lifecycle methods are onInit(), render(), onDispose(), plus the property-pane hooks onPropertyPaneFieldChanged(), getPropertyPaneConfiguration(), and onPropertyPaneConfigurationComplete().

In practice:

  • onInit() — fetch the service-context-bound dependencies (PnP setup, MSAL client, AAD token providers, current user context). Return a Promise — SPFx waits for it before render(). This is the one place teams misuse most — they fetch list data here, which blocks first paint.
  • render() — mount React via ReactDom.render() (legacy) or createRoot() in newer SPFx versions. Keep it idempotent — SPFx calls it on every property-pane change.
  • onDispose() — ReactDom.unmountComponentAtNode() to prevent memory leaks. People forget this on extensions and SPAs.

The hook I regret using: onAfterPropertyPaneChangesApplied for triggering side effects like cache invalidation. It ran twice in some browser conditions and produced duplicate API calls. I moved the logic into a debounced React effect tied to the prop values instead.

Q 1.2
What is the difference between reactive vs non-reactive property panes, and when would you choose each?

Reactive (default) re-renders the web part on every keystroke. Non-reactive (disableReactivePropertyChanges = true) only commits when the user clicks Apply.

  • Reactive — fine for cheap toggles, dropdowns, or anything where the user expects live preview (e.g., theme/colour switch).
  • Non-reactive — mandatory when changes trigger expensive operations (API calls, REST searches, large rerenders). I switch to non-reactive whenever a property pane field can produce a network call.
protected get disableReactivePropertyChanges(): boolean { return true; }
Q 1.3
Explain SPFx solution scope: tenant-wide vs site-collection. Where does the bundle physically live, and what are the trust implications?

SPFx packages are .sppkg files uploaded to the App Catalog. Two catalogs exist:

  • Tenant App Catalog — a single SharePoint site collection (one per tenant). If skipFeatureDeployment: true is in package-solution.json, the solution is auto-trusted tenant-wide and the web part appears everywhere without per-site activation.
  • Site Collection App Catalog — enabled per site, useful for ringed rollouts.

Bundle JS lives wherever cdnBasePath in write-manifests.json points — either Office 365 CDN (the default after --ship), a custom Azure CDN, or the solution's bundled assets in ClientSideAssets.

Trust implication: tenant-scoped solutions with API permissions need explicit approval in the SharePoint Admin Center → API Access. I once shipped a Graph-calling web part to UAT and forgot this — the calls returned 403 until the admin approved the permission request.

Q 1.4
How do you call a custom AAD-protected Web API from SPFx without exposing client secrets?

Use AadHttpClient from @microsoft/sp-http with a pre-approved API permission in package-solution.json:

// package-solution.json
"webApiPermissionRequests": [
  { "resource": "contoso-api", "scope": "user_impersonation" }
]

// In your web part
const client = await this.context.aadHttpClientFactory.getClient("api://contoso-api");
const resp = await client.get(url, AadHttpClient.configurations.v1);

The token is acquired silently via the SharePoint OAuth flow — no client secret ever lives in code. The admin grants consent once in SP Admin Center → API Access.

Anti-pattern I have seen teams hard-code client_secret values into SPFx code "because we couldn't get AadHttpClient to work." That secret then sits in CDN-cached JavaScript. Never. Use a backend Function with managed identity if you need a confidential flow.
Q 1.5
What are the three extension types and when do you reach for each?
  • Application Customizer — header/footer/global script injection. Use for tenant branding, global notification banners, or analytics injection. Scoped to site or list.
  • Field Customizer — render a single column's cell in a list view. Use for status pills, conditional formatting that exceeds JSON column formatting capability, or anything needing live calculation.
  • ListView Command Set — adds toolbar buttons + ECB items on a list. Use for bulk actions, "open in approval workflow," external system pushes.

Plus Form Customizer (added in SPFx 1.15) — replaces the entire New/Edit/Display form for a content type. This is the one you reach for when migrating AngularJS custom list forms — directly relevant to this role.

Q 1.6
How do you implement isolated permissions for an SPFx solution? Why might you NOT want isolation?

Set isDomainIsolated: true in package-solution.json. SPFx then loads the web part inside a unique contoso.sharepoint.com-style iframe domain, separating its OAuth permissions from the rest of the page.

Pro: if the web part has elevated Graph permissions (e.g., Mail.ReadWrite), other web parts on the page cannot piggyback off that token.

Con: iframe communication is async; you lose direct DOM/sibling-WP access, theme tokens behave oddly, and dev-server hot-reload becomes painful. For most departmental web parts I leave it off — turn it on only when you have sensitive APIs.

Q 1.7

Q 5.4
What's the role of Strict Contextual Escaping (SCE) in AngularJS and what's the React equivalent?

AngularJS uses $sce.trustAsHtml() to render trusted HTML. React's default is to escape all string content. The equivalent is dangerouslySetInnerHTML={{ __html: trustedHtml }} — explicitly named to make developers think twice.

Migration risk: anywhere the old code used $sce.trustAsHtml, the original developer either had a legitimate reason (admin-curated HTML) or a security hole. Always re-validate.

Do Sanitize all user-generated HTML through DOMPurify before rendering with dangerouslySetInnerHTML. Treat the existence of either function as a security review trigger.
Q 5.5
Common AngularJS-to-React migration pitfalls — top 5.
  1. Re-render frequency — Angular dirty-checks on every digest cycle (~every 200ms), so devs got lazy with reference equality. In React, mutating state arrays/objects without copying breaks re-renders silently.
  2. Promise behavior — Angular's $q integrates with the digest cycle so promise resolution triggers re-render. Native Promises don't — must call setState.
  3. Event timing — ng-click ran inside the digest cycle. React event handlers don't. Code that assumed "after click, scope is updated" needs useEffect.
  4. Two-way binding muscle memory — developers will instinctively write uncontrolled components or skip onChange handlers. Set up ESLint rules to flag uncontrolled components on form fields.
  5. jQuery dependencies — many AngularJS apps had jQuery globally available. Banish it in SPFx — React + Fluent UI handles every DOM need.
Q 5.6
How do you handle the situation where the legacy AngularJS app uses $resource for a custom WCF service?

Three sub-questions:

  • Is the WCF service still needed? Often it's exposing data that's now in SharePoint Online or Dataverse — remove the dependency entirely.
  • If kept, modernize the service — wrap as an Azure Function or App Service with OpenAPI; build a Custom Connector or call directly via AadHttpClient.
  • Replace $resource with a typed client — generate a TypeScript client from OpenAPI (openapi-generator) and use it from a custom hook. Migrate one $resource at a time.
Q 5.7
When migrating, do you re-write everything at once or run AngularJS and React side-by-side?

For anything beyond ~5 forms or one app, never big-bang. The hybrid approach:

  • Wrap legacy AngularJS pages in iframes embedded in SPFx web parts during transition — preserves user access while you rebuild.
  • OR mount the new React app as a child of the AngularJS app using a single-spa-style adapter (rare; usually iframe is simpler in SharePoint).
  • Migrate page by page, route by route. Track progress against a backlog of "screens to replace."
  • Once last route is replaced, remove the iframe wrapper.
Q 5.8
How do you migrate AngularJS forms with complex validation rules to React?

AngularJS forms use ng-required, ng-pattern, custom $validators. In React I always reach for react-hook-form with zod or yup for the schema:

const schema = z.object({

  budget: z.number().positive(),

Q 5.9
How do you handle global state that AngularJS used $rootScope for?

Options ranked by appropriateness:

  1. React Context + useReducer — for small global state (current user, theme, feature flags). Built-in, zero deps.
  2. Zustand — for medium app state. Lightweight (1 KB), no boilerplate.
  3. Redux Toolkit — only when you genuinely need time-travel debugging, complex middleware, or a large team. Overkill for most SPFx web parts.
  4. React Query / TanStack Query — for server state (lists, items, user profile). This is actually most of what $rootScope held in old SP apps. Use it religiously.

Rapid-Fire & Advanced Edge Cases

Twenty short questions for fast disqualification. Either the candidate knows or they don't.
Maximum size of a single SPFx .sppkg?
25 MB recommended, hard ~100 MB. Beyond that the upload to App Catalog can fail.
What is domElement in an SPFx web part?
The DOM node SPFx provides for your web part to render into. You mount React inside it via ReactDOM.render(element, this.domElement).
Difference between useState and useRef in terms of re-renders?
useState triggers a re-render on update. useRef does not — it persists a value across renders without triggering one. Use refs for DOM nodes, timers, instance values.
SPFx hosted workbench URL?
https://<tenant>.sharepoint.com/_layouts/15/workbench.aspx for hosted; https://localhost:4321/temp/workbench.html for local.
Why doesn't local workbench support API permissions?
Local workbench has no SharePoint context, so it can't issue AAD-mediated tokens. Test API calls in hosted workbench or a real page.
Throttling: what HTTP status does SharePoint return?
429 (Too Many Requests) and sometimes 503. Always check the Retry-After header value and honor it.
Max list view threshold in SPO?
5,000 items per view query. The list can hold 30 million items, but a single query can't return more than 5K without indexed columns and proper paging.
Power Automate "Get items" default item limit?
100 by default. Set to up to 5,000 with pagination enabled. Beyond 5,000 you must page manually.
Power Automate concurrency upper limit on Apply to Each?
50 parallel branches.
Power Automate flow run history retention?
28 days for runs, plus configurable longer if using Solutions + Dataverse.
What's the difference between .then() and await in PnP JS?
Functionally equivalent. await requires the containing function to be async. PnP queryables also support .then() implicitly because they're Thenable.
SPFx solution-level isInvokableByOuterApp — what is it?
Allows the web part to be invoked from another web part / framework (Teams, Viva). Used for the SharePoint-as-service-provider pattern.
Default React version in SPFx 1.18+?
React 17. SPFx 1.19+ moves to React 17.0.1 with TypeScript 4.7. SPFx is conservative on React major versions.
Difference between SPFI and spfi() in PnP JS v3?
PnP v3 dropped the global sp instance. You now create a factory per context: const sp = spfi().using(SPFx(this.context)). Allows multi-context apps without singleton pollution.
Why use Fluent UI instead of Material UI in SPFx?
Fluent UI is Microsoft's design system; it matches SharePoint native UI and inherits the SPFx theme automatically. Material UI would look out of place.
What is @microsoft/teams-js for in SPFx?
Lets an SPFx web part also run as a Teams tab. The Teams context provides theme, locale, user info via that SDK.
How do you call a Power Automate flow from SPFx?
Two options: HTTP-triggered flow with AAD-protected endpoint (call via AadHttpClient), or a Power Automate "Manually triggered" flow + button click that posts to its public URL with a shared secret in body.
SharePoint webhook expiration?
Max 6 months from creation. Must be renewed programmatically before expiry, ideally on a daily Azure Function timer.
Difference between Microsoft Graph and SharePoint REST for getting list items?
Graph has unified throttling and a cleaner shape but lacks some SharePoint-specific features (managed metadata as native types, taxonomy navigation). SharePoint REST gives full fidelity. Use Graph when crossing SharePoint + other M365; use SP REST for deep SharePoint scenarios.
In Power Automate, what is "Run-only users" for a flow?
Users who can trigger an instant flow but not edit it. Used for sharing automation with end-users while keeping the flow definition admin-controlled.

export function useSharePointList<T>(
  context: WebPartContext,
  listName: string,
  selectFields: string
) {
  const [data, setData] = useState<T[]>([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<string | null>(null);
  const [tick, setTick] = useState(0);

  useEffect(() => {
    let cancelled = false;
    setLoading(true);
    setError(null);
    const sp = spfi().using(SPFx(context));
    sp.web.lists.getByTitle(listName).items.select(selectFields)()
      .then(d => { if (!cancelled) { setData(d); setLoading(false); } })
      .catch(e => { if (!cancelled) { setError(e.message); setLoading(false); } });
    return () => { cancelled = true; };
  }, [listName, selectFields, tick]);

  const refetch = useCallback(() => setTick(t => t + 1), []);
  return { data, loading, error, refetch };
}

Write a TypeScript utility function that retries a failed async call with exponential backoff.
export async function withRetry<T>(
  fn: () => Promise<T>,
  maxAttempts = 4,
  baseDelayMs = 1000
): Promise<T> {
  let attempt = 0;
  while (true) {
    try {
      return await fn();
    } catch (err: any) {
      attempt++;
      if (attempt >= maxAttempts) throw err;
      const isThrottle = err?.status === 429 || err?.status === 503;
      const retryAfter = isThrottle
        ? (parseInt(err?.headers?.get?.("Retry-After") || "0") * 1000)
        : 0;
      const delay = retryAfter || baseDelayMs * Math.pow(2, attempt - 1);
      await new Promise(res => setTimeout(res, delay));
    }
  }
}

Respects SharePoint's Retry-After header for 429 throttling, falls back to exponential backoff for other failures.

TS 13.1
What does strict: true in tsconfig.json enable, and why should every SPFx project have it?

strict: true is a shorthand that enables a bundle of compiler flags simultaneously:

  • strictNullChecks — null and undefined are not assignable to other types. This is the most impactful; it forces you to handle nullable API responses explicitly.
  • noImplicitAny — Variables that cannot be inferred must have an explicit type annotation; no silent any.
  • strictFunctionTypes — Function parameter types are checked contravariantly in assignability.
  • strictPropertyInitialization — Class properties must be initialized in the constructor (or marked ! for definite assignment).
  • noImplicitThis — this in functions must have an explicit type.

In SPFx, strictNullChecks alone catches dozens of bugs that would otherwise only surface when a list item is missing a field or an API returns null. The SPFx Yeoman template turns strict on by default in SPFx 1.15+.

Interviewer tip: Ask the candidate to show a code change they had to make after enabling strictNullChecks. If they can't name one, they've never actually turned it on.
TS 13.2
Explain TypeScript generics with a concrete SPFx data-fetching example.

Generics let you write a function or class that works over a type supplied by the caller, preserving type safety without widening to any.

// Generic fetch wrapper for SharePoint lists
async function getListItems<T>(
  sp: ReturnType<typeof spfi>,
  listName: string,
  selectFields: (keyof T & string)[]

  return sp.web.lists
    .getByTitle(listName)
    .items
    .select(...selectFields)<T[]>();

// Caller is fully typed — no 'any'
interface IProject {
  Id: number; Title: string; Status: string;
}
const projects = await getListItems<IProject>(sp, "Projects", ["Id","Title","Status"]);

The generic parameter T flows through: the return type is T[], so projects[0].Status has autocomplete and compile-time checking.

TS 13.4
What are type guards and how do you write a custom one?

A type guard is an expression that narrows a type within a conditional block. TypeScript provides built-in guards (typeof, instanceof, in) and lets you write user-defined ones using the is predicate syntax.

// Built-in guards
if (typeof value === "string") { /* value is string here */ }
if (error instanceof Error) { /* error.message is available */ }
if ("Id" in item) { /* item has Id property */ }

// User-defined type guard — useful for discriminating API responses
interface ITask { kind: "task"; Title: string; AssignedToId: number }
interface IProject { kind: "project"; Title: string; Budget: number }

function isTask(item: ITask | IProject): item is ITask {
  return item.kind === "task";
}

function renderItem(item: ITask | IProject) {
  if (isTask(item)) {
    // TypeScript knows item is ITask here
    console.log(item.AssignedToId);
  }
}
TS 13.5
What is a discriminated union and how does it improve React state modeling in SPFx?

A discriminated union is a union type where each member has a common literal property (the "discriminant") that uniquely identifies it. TypeScript uses the discriminant to narrow types in switch/if blocks.

// Instead of: isLoading: boolean, error: string | null, data: T[]
// which allows illegal states like isLoading=true AND data=[...]

type FetchState<T> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T[] }
  | { status: "error"; message: string };

const [state, setState] = useState<FetchState<ITask>>({ status: "idle" });

// In JSX — exhaustive and type-safe
switch (state.status) {
  case "loading": return <Spinner />;
  case "error":   return <MessageBar>{state.message}</MessageBar>;
  case "success": return <TaskList items={state.data} />;
  default:        return null;
}

Discriminated unions eliminate impossible state combinations at the type level — you can't accidentally render a list while also showing a loading spinner.

TS 13.6
Explain interface vs type — when do you use each in SPFx?
  • interface: Preferred for object shapes (SharePoint list item models, component props, service contracts). Supports declaration merging — useful when augmenting third-party types. Can be extended with extends.
  • type: Preferred for unions, intersections, mapped types, conditional types, and aliases for primitives. Cannot be re-opened/merged.
// interface — list item shape (extensible)
interface IListItem { Id: number; Title: string }

  | "FieldCustomizer"
  | "ListViewCommandSet"
  | "FormCustomizer";

TS 13.7
What is unknown and why is it safer than any? Give an SPFx example.

any disables type checking entirely — you can call methods on it, pass it anywhere, and TypeScript won't complain. unknown is the type-safe alternative: it accepts any value, but you must narrow it before doing anything useful with it.

// Catching errors — before TypeScript 4.0 this was 'any'
try {
  await sp.web.lists.getByTitle("Tasks").items();
} catch (err: unknown) {
  // Cannot access err.message directly — must narrow first
  if (err instanceof Error) {
    setError(err.message);
  } else if (typeof err === "string") {
    setError(err);
  } else {
    setError("An unknown error occurred");
  }
}

Use unknown for catch variables, JSON.parse results, and any value coming from an external API until you've validated its shape.

How do you use useReducer with TypeScript to manage a complex form state in a Form Customizer?
interface IFormState {
  title: string;
  status: string;
  priority: number;
  errors: Record<string, string>;
  saving: boolean;
}

type FormAction =
  | { type: "SET_FIELD"; field: keyof IFormState; value: string | number }
  | { type: "SET_ERRORS"; errors: Record<string, string> }
  | { type: "START_SAVE" }
  | { type: "SAVE_DONE" };

    case "SET_ERRORS":

    case "START_SAVE":
      return { ...state, saving: true, errors: {} };
    case "SAVE_DONE":
      return { ...state, saving: false };

}

SC 24.1
SCENARIO — Permissions + Large Data: A client has a SharePoint list with 400,000 expense records. Every item must be visible only to the employee who created it and their manager chain. Users complain queries are slow and some get a "list view threshold" error. How do you redesign this?

Why this is a trap: The instinct is to break inheritance on 400,000 items. That is catastrophically wrong. SharePoint's permission model stores a separate ACL row per uniquely-permissioned item. At 400,000 items, you've created 400,000 SQL ACL entries — the site will crawl and the permission check on every query will time out.

Correct architecture:

  1. Never use item-level unique permissions at this scale. Remove them entirely.
  2. Redesign the data access pattern:
    • Add an indexed EmployeeEmail column to the list
    • Every query from the app must filter EmployeeEmail eq 'currentUser@contoso.com' — the index makes this sub-threshold even on 400K items
    • The manager view uses a separate filtered query: ManagerEmail eq 'manager@contoso.com' — also indexed
  3. Enforce access in the application layer, not at SharePoint's permission layer — every API call includes the current user's UPN in the filter; users cannot remove this filter because it's enforced server-side in an Azure Function middleware that wraps the SharePoint REST call
  4. If regulatory isolation is truly required — move the data to Dataverse (no row-level threshold, native row-level security) or Azure SQL with row-level security policies
Red flag answer "Break inheritance on each item and set permissions using Power Automate when a new item is created." This scales to about 5,000 items before it destroys performance. Reject this answer.
SC 24.2
SCENARIO — SPFx Performance: Your web part works perfectly in the SPFx workbench and on a blank SharePoint page. But when it's placed on a page that already has 7 other web parts, the browser tab freezes for 4–5 seconds on load. Users are complaining. Walk through your diagnosis and fix.

Step 1 — Isolate the cause. Open browser DevTools → Performance tab → record the page load. Look for:

  • Long tasks (red triangles) — anything over 50ms blocking the main thread
  • Multiple identical network requests — 8 web parts each making their own call to the same list? That's 8 throttleable requests firing simultaneously
  • Bundle size — check the Network tab for your web part's JS bundle. If it's 2MB+, the parse/compile time alone causes freezing

Most common root causes and fixes:

  • Bundle too large: You've imported a full library (Lodash, Moment, a chart library) without tree-shaking. Fix: use targeted imports (import debounce from 'lodash/debounce'), externalise via config.json CDN, or replace with lighter alternatives
  • 8 parallel API calls to the same list: The ServiceScope pattern fixes this — register a single data service in onInit() shared across all web parts. One call, cached result, all web parts consume it
  • Synchronous work in onInit: If you're doing heavy computation (sorting, transforming 5,000 items) synchronously, it blocks the browser. Move to a Web Worker or defer with setTimeout(fn, 0) to yield to render
  • React rendering storm: Each web part's state update triggers a re-render of its children. If children are expensive (deep trees, no memoization), it cascades. Fix: React.memo, useMemo, useCallback on stable references
Impress answer Mention that you've used the Chrome Performance tab's "Bottom-up" view to identify which specific function is eating the main thread, not just guessing.
SC 24.3
SCENARIO — Power Automate Scale: You need to build a flow that processes 50,000 SharePoint list items overnight — calling an external API for each item. The external API has a rate limit of 200 calls per minute. The flow must complete before 6 AM. How do you architect this without timeouts or rate-limit errors?

Why a naive "Apply to each" fails: Apply to each with 50,000 items will hit Power Automate's 30-day run limit. Each iteration that takes even 1 second means 50,000 seconds = 13.8 hours. Plus the rate limit will be violated immediately if concurrency is high.

Correct architecture:

  1. Chunked child flows: Split items into batches of 500. A parent scheduled flow reads the list using getPaged() (500 items per page) and triggers a child flow per batch. Child flows run in parallel (up to concurrency limit of the Apply to Each).
  2. Rate-limit compliance inside the child flow: 
    Apply to each item in batch:
  → Call external API
  → Delay: 300ms  ← ensures max 200 calls/min within a single flow instance
  3. Concurrency tuning: With 100 parallel child flows each processing 500 items at 300ms delay = each child takes ~2.5 minutes. 100 batches × 2.5 min = 250 minutes ≈ 4 hours. Fits the window.
  4. Error handling per item: Wrap each API call in a Scope with "Run after: Has failed" to log failures to a separate SharePoint error log list — don't let one bad item kill the whole batch.
  5. Idempotency: Add a ProcessedOn date column to each item. Skip items where this column is already set — so if the flow is re-run, it only processes the ones that failed.
Red flag answer "Just use Apply to each with concurrency set to 50." This ignores the rate limit, the timeout window, and error recovery. The flow will randomly fail at 2 AM with no way to resume.
SC 24.4
DEEP CONCEPT — React Hooks: A junior developer wrote this component. Users report it makes API calls indefinitely even after the data loads, causing hundreds of network requests. Find the bug and explain exactly why it happens. 
function ProjectList() {
  const [items, setItems] = useState([]);

  useEffect(() => {
    sp.web.lists.getByTitle("Projects").items()
      .then(data => setItems(data));
  }, [items]);  // <— dependency array

  return <ul>{items.map(i => <li key={i.Id}>{i.Title}</li>)}</>;
}

The bug: items is in the dependency array of useEffect. This creates an infinite loop:

  1. Component mounts → useEffect runs → API call fires
  2. API returns data → setItems(data) is called → React sets a new array reference (even if data is identical)
  3. React compares the dependency: old items reference !== new items reference → dependency changed → useEffect runs again
  4. Goto step 2. Infinitely.

Why a new array is always a new reference: JavaScript arrays are compared by reference, not by value. [] === [] is false. Every setItems(data) call creates a new array object in memory, so React always sees the dependency as "changed."

Fix — empty dependency array (run once on mount):

useEffect(() => {
  sp.web.lists.getByTitle("Projects").items()
    .then(data => setItems(data));
}, []);  // ← empty array = run once on mount only

Fix — with cleanup for cancelled requests (production-grade):

useEffect(() => {
  let cancelled = false;
  sp.web.lists.getByTitle("Projects").items()
    .then(data => { if (!cancelled) setItems(data); });
  return () => { cancelled = true; };  // cleanup: prevents state update after unmount
}, []);
Follow-up Ask: "What happens if a user navigates away before the API call completes?" — The answer is a React warning: "Can't perform a React state update on an unmounted component." The cleanup function above prevents it.

DEEP CONCEPT — AngularJS $watch vs React useEffect: You're migrating an AngularJS controller that uses $scope.$watch('project', handler, true) — the third argument true enables deep equality comparison on a complex nested object. How do you replicate this in React without causing an infinite render loop?

What AngularJS's deep watch does: It traverses the entire object tree on every digest cycle and fires handler only when any nested value actually changed (by value, not reference). It is expensive but prevents false positives.

Why useEffect([project]) fails: React compares dependencies by reference (Object.is). If project is an object created fresh on each render (e.g., from a selector or inline creation), the reference changes every cycle → infinite loop.

Solution 1 — JSON.stringify comparison (simple, works for serialisable objects):

const projectRef = useRef(project);
useEffect(() => {
  if (JSON.stringify(project) !== JSON.stringify(projectRef.current)) {
    projectRef.current = project;
    handler(project);
  }
});  // No dep array — runs every render, but only fires handler on actual change

Solution 2 — useMemo to stabilise the reference:

const stableProject = useMemo(() => project, [project.id, project.status, project.name]);
// Now useEffect only fires when id, status, or name actually change
useEffect(() => { handler(stableProject); }, [stableProject]);

Solution 3 — useDeepCompareEffect (third-party hook or custom):

function useDeepEffect(fn, deps) {
  const ref = useRef(deps);
  if (!deps.every((d, i) => JSON.stringify(d) === JSON.stringify(ref.current[i]))) {
    ref.current = deps;
  }
  useEffect(fn, [ref.current]);
}
Key insight to demonstrate React deliberately does not do deep equality — it is a performance choice. Deep comparison on every render defeats React's optimization model. The correct answer is to restructure state so that the relevant primitive values are the dependencies, not the whole object.
