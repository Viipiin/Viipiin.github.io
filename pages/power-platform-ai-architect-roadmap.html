📅 Study Plan — 90-Day Sprint & 180-Day Roadmap

Two-phase plan. The first 90 days close the architect-skill gap (design patterns, integration, AI). The next 90 days build a portfolio, drive certifications, and start positioning for senior roles. Plan assumes ~2 hrs/weekday + 5 hrs/weekend = ~15 hrs/week.

How to use this plan: Pick the next uncompleted week, block the time on your calendar, and treat the "Build" task as non-negotiable. Theory without code does not land architect interviews — every week ends with a deployable artifact you can put in your portfolio.

  • Network: deploy with Private Endpoints, disable public network access, route through APIM for auditing
  • India data residency: Azure OpenAI in South India region (added 2024); fall back to Sweden Central / East US 2 for newest models

▸ MCP — What Architects Must Know

Model Context Protocol — In One Page

  • What it is: Open standard for connecting LLMs/agents to external tools, data, and prompts. Think "USB-C for AI."
  • Why it matters: Without MCP, every agent framework reinvents tool-calling. With MCP, your agent can use any of 6,400+ existing servers and your own.
  • Server types: stdio (local subprocess), Streamable HTTP (network), with capabilities for tools / resources / prompts.
  • Build one in a day: Node/TypeScript or Python SDK. Wrap your Dataverse Web API, your APIM, your SAP connector as MCP servers — any agent can then use them.
  • Security: Treat MCP servers as untrusted code. Use principle of least privilege, audit tool calls, never auto-approve destructive operations.
  • Architect interview question incoming: "How would you let Copilot Studio agents call your enterprise's APIs safely?" Answer: MCP servers fronted by APIM with per-tool RBAC.

Interview Questions — Architect Level (Real Ones)

Q1. We have 80 million records in a Dataverse table and queries are slow. What's your plan?
Talk about: query patterns, indexes, alternate keys, partition strategy (is it an elastic table candidate?), archiving cold data to Synapse Link / Blob, virtual tables for read-only history, optimizing FetchXML/OData filters, table-level audit impact. Always end with "I'd want to see a query plan first."
Q2. A plugin keeps timing out. Walk me through how you'd fix it.
Sandbox is 2 min. Options: convert to async plugin step, move heavy work to Azure Function via Service Bus, batch the calls, optimize SDK queries (avoid RetrieveAllEntities), avoid loops, paginate. Talk about observability — App Insights, plugin trace logs.
Q3. Should this be a Canvas app or Model-driven?
Don't pick blindly. Ask: who's the user? What's the data shape? What's the device? Is offline needed? Is security row/column complex? Then use the decision matrix to defend your call.
Q4. Tell me about an architect decision you regretted.
Have one ready. Pick a real case: "I chose Power Automate over a Function for X — it worked initially but throughput hit a wall at 50k/day. Refactored to Function + Service Bus. Lesson: model the load before picking the tool."
Q5. How would you design a Copilot bot answering from 10,000 policy documents with PII?
RAG architecture (chunking, embeddings, vector store), AI Search hybrid, citations enforced, PII redaction at input via AI Language, content safety on output, audit logging of every prompt+response, FinOps with per-team chargeback, governance approval through Purview. Don't say "I'd use ChatGPT."
Q6. The customer wants us to skip ALM "to save time." How do you respond?
Acknowledge pressure → quantify future cost (lost hours per hotfix, audit risk) → propose a minimal viable ALM (Git + 2 environments + automated solution import) → never say "no, we must" — say "let me show you a 2-day setup that pays back in week 2."
Q7. How do you decide between Service Bus, Event Grid, and Event Hubs?
Commands & ordered messages → Service Bus. Reactive events / fan-out to many handlers → Event Grid. Telemetry / massive throughput / replay → Event Hubs. Have one real story for each.
Q8. We have a $40k/month Azure OpenAI bill. Reduce it.
Switch eligible workloads to GPT-4o-mini, enable prompt caching, shorten system prompts, downsize embedding dimensions, batch where possible, evaluate PTU, set hard budgets, dedupe similar queries via cache. Track $/answered query.
Q9. Multi-environment strategy for 4 business lines on one tenant?
Default + 1 dev per BL + shared sandbox + UAT + Prod per BL OR shared Prod with environment groups. Discuss capacity, DLP policies per env, environment routing, ALM pipeline reuse via templates.
Q10. How would you embed an agent inside Microsoft 365 Copilot?
Build with Copilot Studio (declarative or custom-engine agent), package as Teams app, publish via admin center, govern via Purview, monitor via Microsoft 365 Copilot Analytics. Discuss declarative (no-code, hosted by MS) vs custom-engine (your own LLM).

🧩 Architect Thinking Framework

The transition from senior developer to architect is mostly a thinking change. These are the mental models that separate the two.

The Architect's Default Loop

  1. Clarify constraints before picking solutions. (Budget? Users? SLA? Compliance? Skill in the team?)
  2. Generate 2–3 options, not one. If you only see one option, you're a developer; an architect sees a space.
  3. Make trade-offs explicit. "Option A is cheapest, B is fastest to build, C scales best — I recommend B because we have a 4-week deadline and can refactor in Q3."
  4. Document the decision (ADR — Architecture Decision Record). Future you will thank present you.
  5. Plan for change. What if the assumptions break in 6 months?

Decision-Making Frameworks

1. Well-Architected (5 pillars)

Reliability, Security, Cost, Operational Excellence, Performance. Map every design choice against these.

2. RACI for stakeholders

Who's Responsible / Accountable / Consulted / Informed for each architecture domain (security, data, integration, AI). Stops the "everyone in every meeting" failure.

3. Build vs Buy vs SaaS

Always evaluate the third option. SaaS often beats both for non-differentiating features.

4. Cost / Effort / Value matrix

2×2 grid: high-value / low-effort first, low-value / high-effort never. Use to prioritize a backlog of architectural improvements.

5. CAP & PACELC

For data design — what consistency are you giving up for availability? PACELC adds latency vs consistency for the non-partitioned case.

6. ATAM (Architecture Tradeoff Analysis)

Lightweight: list quality attributes (perf, sec, mod) → rank → identify scenarios that stress each → see where current arch breaks.

7. STRIDE (security threats)

Spoofing, Tampering, Repudiation, Info disclosure, DoS, Elevation. Walk through these for any integration design.

8. C4 Model for diagrams

Context → Container → Component → Code. Most architects only draw the first two; that's correct — go deeper only when needed.
