The Three Root Causes of Licensing Overspend

🔌

Wrong Connector Choices

One premium connector in a flow forces premium licensing for every user who runs it. The single most common unexpected cost driver.

🗃

Unnecessary Dataverse

Dataverse is often the right choice — but using it for simple list-style data when SharePoint would suffice adds avoidable per-user premium cost.

🏗

Architecture Without Licensing Awareness

Elegant architectures built without licensing awareness force premium patterns that simpler, well-designed alternatives would avoid entirely.

Right-Sizing is a Design Activity

Licensing optimization is not a procurement exercise. It happens at the architecture table, before the first line of code is written. The correct question is not "can we afford this license?" — it is "does this architectural choice justify this license?"

A properly designed Power Platform solution fits the licensing model to the requirement. A poorly designed one forces premium licensing on the cheapest possible requirements, then blames Microsoft pricing for the bill.

Section 3

What Actually Increases Cost in Power Platform?

Understanding the cost drivers before making architecture decisions is the most reliable way to avoid surprise licensing bills.

Licensing Validation Required Prices and entitlements change. Validate all licensing specifics against the Power Platform pricing page and the Power Platform Licensing Guide (January 2026) before presenting to clients.

1. Premium Connectors — The Cascade Effect

The most common licensing trap. Any flow or app that uses even one premium connector requires a premium license for all users who interact with it. This is the cascade: one premium connector in a shared flow means every user of that flow needs a premium entitlement.

ExamplesTypeLicense Required
SharePoint, Teams, Outlook, OneDrive, Planner, Forms, Excel, Approvals, NotificationsStandardIncluded in M365 / base licenses — no premium needed
SQL Server, Dataverse, HTTP, ServiceNow, Salesforce, Azure Blob Storage, Adobe Sign, DocuSignPremiumRequires Power Apps/Automate Premium, per-app, or per-flow license
Any connector you build yourselfCustomTreated as premium — same licensing implications
Verify Connector Tier Full standard connector list: learn.microsoft.com/standard-connectors — Full premium list: learn.microsoft.com/premium-connectors. These lists update; verify before finalizing architecture.

2. Dataverse as the Backend

Dataverse requires a Power Apps Premium license (or a Dynamics 365 license that includes Dataverse access). If you build a Canvas App backed by Dataverse and assign it to 50 users, all 50 need premium — even if the app is simple. This is not a reason to avoid Dataverse; it is a reason to confirm that Dataverse's capabilities (relational model, field-level security, advanced audit, business rules, plugins) are actually required before introducing it.

3. Custom Connectors Are Premium

Building a custom connector to avoid a specific premium connector does not save on licensing. Custom connectors are treated as premium and still require premium entitlement for all users. The primary benefit of a custom connector is API contract stability, not licensing reduction.

4. Managed Environments

Managed Environments unlock important governance features: solution checker enforcement, sharing controls, IP firewall, DLP policies, usage insights. However, they require that all active users in the environment have a qualifying premium license or a Managed Environments add-on. Enabling Managed Environments across all environments — including dev — before confirming license coverage creates compliance gaps. Source: Managed Environments licensing.

5. Per-User vs Per-App — The Over-Buy Problem

Power Apps per-user grants unlimited apps. Power Apps per-app grants access to one specific app. For users who only ever need a single app, per-app is significantly cheaper. Organizations that blanket-assign per-user without analyzing actual app usage per user are a very common source of avoidable spend.

Architecture Action During design, map each user role to the number of Power Platform apps they need. Users needing only one app → per-app. Users with access to multiple apps → per-user may become cost-effective at around 3+ apps per user. Validate the crossover point with current official pricing.

6. Unattended / Process Automation

Power Automate flows that run without human interaction (unattended RPA, scheduled flows using premium connectors) may require a per-flow license (a flat fee per flow per month). For high-volume automation with many flows, per-flow can be cost-effective vs per-user. This comparison requires a concrete flow count and user count — not a rule of thumb.

7. Environment Sprawl

Every Dataverse-enabled environment consumes Dataverse storage capacity. Unmanaged environments accumulate storage charges and introduce governance overhead. They also multiply: development environments become UAT sandboxes for other teams until storage costs appear on billing with no clear owner.

8. The HTTP Connector

The HTTP connector in Power Automate is premium. Any flow that makes a raw API call to an external service requires premium licensing for every user of that flow. This is consistently the least-expected cost driver in integration architectures. Evaluate whether a Logic App, APIM policy, or Azure Function can sit between Power Automate and the external API before defaulting to the HTTP connector.

9. Orphaned Flow Ownership

Flows owned by users who leave an organization stop running and become orphaned at account deactivation. Emergency license assignments or manual ownership transfers at offboarding carry real operational cost — and these incidents never appear in license budget forecasts. They appear in incident tickets on a Monday morning.

Decision Framework: Cheapest Valid Option First

Before selecting any technology, connector, or data store, work through this decision tree. The principle: start with what M365 already gives you, escalate only when a genuine requirement forces it. "Architecturally elegant" is not a justification for premium. Business need is.

Core Principle Do not force premium just because it is the cleaner solution. Premium is justified when standard capabilities create real operational, security, or scalability risk — not merely because Dataverse or premium connectors feel more robust.

The Decision Tree

Q1 — Can this requirement be fully met with Microsoft 365 included capabilities?
YES →
✓ Use SharePoint, Teams, Outlook, Forms, Planner, OneDrive, Excel Online, Approvals. No premium license required. Stop here.
NO → Continue to Q2
Q2 — Is the data requirement simple enough for SharePoint Lists or Microsoft Lists?
✓ YES — <30M items, no complex relational model, no field-level security, no plugins needed → Use SharePoint/Lists. Pair with standard connectors. No premium.
✗ NO — Need relational model, row-level security, complex audit, or >30M rows → Dataverse is justified. Premium license required.
Q3 — Does the integration require a premium connector or the HTTP connector?
✓ NO — All connectors used are on the standard connector list → Standard license sufficient.
✗ YES — First ask: can the premium connector be wrapped in an Azure Function or Logic App behind a standard trigger? If still unavoidable → premium required for all users of that flow/app.
Q4 — Is this solution being built inside a Dynamics 365 context where licenses already exist?
✓ YES — Many Dynamics 365 licenses include Power Apps and Dataverse use rights. Verify entitlements for your specific D365 license tier → premium may already be covered.
✗ NO — Continue to per-user vs per-app analysis.
Q5 — Do all users need access to multiple premium Power Platform apps?
YES (3+ apps) → Per-user license is likely more economical. Calculate the crossover point with current pricing.
NO (1–2 apps) → Per-app license is almost always more economical per user. Do the math before purchasing.

Section 5

Standard vs Premium vs Custom Connector Strategy

Connector selection is the single highest-leverage licensing decision at the solution design stage. One wrong connector choice can turn a standard-licensed solution into a premium-licensed one for all users.

The Fundamental Rule

A Power Apps canvas app or a Power Automate flow takes the highest licensing tier of any single connector used in it. A flow using 10 standard connectors and 1 premium connector is a premium flow for all intents and purposes. There is no blending: the licensing tier is determined by the most premium component present.

TierDefinitionLicense NeededExample Connectors
Standard Connectors included in Microsoft 365 and base Power Platform entitlements M365 license (or base Power Apps/Automate seeded license) SharePoint, Teams, Outlook, OneDrive, Forms, Planner, Excel Online, Approvals, Notifications, Power BI, Azure DevOps (basic), MSN Weather
Premium Connectors that require a standalone Power Apps / Power Automate Premium license Power Apps Premium, Power Automate Premium, per-app, or per-flow SQL Server, Dataverse, HTTP, Azure Blob Storage, Azure Service Bus, ServiceNow, Salesforce, Zendesk, Adobe Sign, DocuSign, Stripe, SendGrid, GitHub (premium)
Custom Connectors you build yourself to connect to APIs not covered by built-in connectors Treated as premium — same requirements as premium connectors Your internal APIs, third-party SaaS APIs not covered by existing connectors, internal microservices

Red-Flag Connector Patterns

  • HTTP connector in any user-triggered flow — forces premium for all users of that flow; evaluate per-flow license or rerouting via Azure Functions
  • Dataverse connector in a simple list app — if the data has no relational requirements, SharePoint often suffices at zero additional license cost
  • Mixing premium and non-premium connectors in shared flows — the single premium connector escalates the entire flow's licensing requirement for all users
  • Using a premium connector in a flow that triggers hundreds of users — validate whether a per-flow license would be more economical than per-user for all 100+ users

Section 7

ALM & Deployment Strategy: Power Platform Pipelines vs Azure DevOps

Both Power Platform Pipelines and Azure DevOps Build Tools are supported ALM paths. Neither is universally better. The right choice depends on team composition, governance requirements, and the complexity of the solution.

Power Platform Pipelines — When They Are Enough

  • Citizen developer or maker-led projects where the team has no Azure DevOps expertise
  • Straightforward source → test → production deployment without conditional logic or external approvals
  • Solutions without code components (no plugins, PCF, Azure Functions as part of the release)
  • Small teams (<5 makers) who need a governed promotion path without full DevOps overhead
  • Organizations that want Power Platform admins to own the pipeline configuration without developer involvement
Important Prerequisite Power Platform Pipelines require a Dataverse database in the host environment. The pipeline host environment must have a Dataverse database provisioned — this consumes Dataverse storage capacity. Confirm that this storage is available before committing to Pipelines as your ALM strategy. Source: Set up pipelines in Power Platform.

Azure DevOps Build Tools — When They Are Better

  • Professional developer teams with existing Azure DevOps investment and expertise
  • Solutions containing code components: plugins, custom PCF, Azure Functions, .NET assemblies
  • Multi-repo, multi-solution scenarios requiring conditional logic, branching strategies, or multi-stage pipelines
  • Enterprise compliance requirements for full pipeline audit, SAST/DAST integration, or custom approval gates
  • Solutions that integrate Power Platform deployment with Azure infrastructure deployment (infra + app in one pipeline)
  • Scenarios where automated solution checker enforcement before deployment is required

Identity Strategy: Service Account vs Service Principal vs Application User vs Managed Identity

The identity used to run automation and integrations has direct implications for security, governance, licensing, and operational resilience. This is one of the most commonly underspecified architectural decisions in Power Platform projects.

Identity TypeWhat It IsLicense NeededBest UseAvoid When
Human-owned account (user) A named user's M365/AAD account owns and runs flows/apps Standard user license Personal flows, development, low-criticality automation Production critical flows — user leaves, flow dies. Shared credentials, governance nightmare.
Service Account A shared named AAD user account (not tied to a person) used for automation Requires a full user license (Power Platform license + M365) Legacy integrations where service principal is not supported; quick setup when SP is unavailable New builds — password rotation risk, MFA complexity, audit ambiguity ("which human did this?"), offboarding risk
Service Principal An app registration in Azure AD with no human identity. Can be granted roles and access permissions. No user license required for Power Automate flows it owns. May need Dataverse application user setup. CI/CD pipelines, Azure DevOps connections, ownership of critical production flows, ALM automation Not supported for all Power Automate trigger types — check supported trigger list. Not ideal for user-delegated flows.
Application User A Dataverse application user mapped to an Azure AD service principal. Provides non-interactive access to Dataverse. No user license required for Dataverse access — application users are non-licensed in Dataverse Backend integrations reading/writing to Dataverse without a human user session, CI/CD, Azure Function-to-Dataverse User-facing flows where delegated identity (acting as the user) is required for row-level security
Managed Identity An automatically managed AAD identity tied to an Azure resource (Function App, Logic App, AKS, etc.). Credentials managed by Azure. No additional license — tied to the Azure resource it runs on Azure-side integrations (Azure Function to Dataverse, Azure Function to SharePoint, Logic App to Power Platform APIs) Direct Power Automate flow ownership — managed identities cannot own Power Automate flows directly (use service principals instead)

The Critical Licensing Note: Service Principals and Flows

Service principals can own Power Automate cloud flows without requiring a per-user Power Automate license. This is officially supported and documented by Microsoft. This makes service principals the correct identity for critical production automation — the flow does not depend on a human user's license remaining active.

Source: Support for service principal owned flows | Manage application users in Power Platform admin center

Best Practice Default Critical production flows should be owned by a service principal, not a named user. This removes the dependency on any individual's license, prevents orphaning when users leave, provides clean audit attribution, and aligns with least-privilege principles.

Identity Selection Decision Tree

Is this automation business-critical or unattended (no human interaction)?
YES →
Is this running inside Azure (Function App, Logic App, AKS)?
YES → Use Managed Identity on the Azure resource. No credential management needed.
NO (Power Automate / Dataverse) → Use Service Principal + Application User. No user license required for Dataverse access.
NO (human-triggered, personal, or low-criticality) →
Will this flow/app persist after the owner potentially leaves?
NO (truly personal, expected lifecycle = that user) → Named user ownership is acceptable.
YES (team or org-wide flow) → Use a service account OR migrate to service principal ownership. Service account is acceptable short-term; service principal is the long-term target.

Setting Up a Service Principal for Power Platform

See Section 12 for practical PowerShell and PAC CLI examples. Official reference: Creating a service principal using PowerShell | Creating a service principal using PAC CLI.

Anti-Pattern Shared service accounts with weak or non-rotating passwords are a security and compliance liability. Service principals with certificate-based authentication (not client secrets) are the secure production pattern. Client secrets expire and rotate; certificates managed in Azure Key Vault are far more robust in enterprise scenarios.

Practical Cost-Saving Patterns

These are concrete, repeatable architectural patterns that reduce licensing cost without compromising quality, governance, or security.

🌞
Pattern 1 — Internal Approval App (M365-Only)

Leave requests, purchase approvals, IT access requests — all using SharePoint + standard connectors + Teams.

When to Use
Internal teams, <30k rows, no relational data, no external integrations
Why It Saves Cost
Zero additional licensing — all M365 included. SharePoint for data, Approvals connector for workflow.
Risks
5,000-item view threshold in SharePoint (manageable with indexed columns). No field-level security.
Governance Note
Add SharePoint site-level permissions. Use Power Platform DLP to restrict connectors to Standard-only for this environment.
📄
Pattern 2 — Departmental Intake App (No Dataverse)

Intake forms for HR, facilities, or support desk. Data stored in Microsoft Lists. Notifications via Teams/Outlook.

When to Use
Single department, form-based data capture, simple reporting via Power BI free
Why It Saves Cost
No premium connectors. Microsoft Lists is included in M365. Reports via Power BI connected to SharePoint data (standard connector).
Risks
Limited reporting capability vs Dataverse native views. No business rules engine.
Governance Note
Document the decision at design review: "Dataverse evaluated and not required because..."
👥
Pattern 3 — Service Principal for Critical Production Flows

Any business-critical unattended flow (nightly sync, order processing, integration) owned by a service principal, not a named user.

When to Use
Production flows that must survive user offboarding; unattended automation; integration flows
Why It Saves Cost
No user license required for SP-owned flows. Eliminates license-at-risk when users leave. Reduces emergency re-assignment incidents.
Risks
Setup complexity higher than user-owned flows. Limited to supported trigger types — verify compatibility.
Governance Note
Register SP in Azure AD. Create application user in Dataverse if Dataverse access needed. Assign least-privilege roles only.
⚙️
Pattern 4 — PP Pipelines Instead of Azure DevOps (Citizen Dev Teams)

Use built-in Power Platform Pipelines for solution promotion instead of standing up Azure DevOps pipelines for simple maker-led solutions.

When to Use
Maker-led projects, no code components, small teams, solutions without CI/CD complexity
Why It Saves Cost
No Azure DevOps setup cost. Admin-controlled in Power Platform admin center. No DevOps agent costs for simple solutions.
Risks
Requires Dataverse database in host environment (storage cost). Limited conditional logic vs full YAML pipelines.
Governance Note
Manage Environments are recommended for the full pipeline feature set but may require premium license coverage for all users.
🔒
Pattern 5 — Application User for Azure-to-Dataverse Integration

Azure Function or Logic App writing to Dataverse using an application user (no interactive user session, no user license consumed).

When to Use
Backend data pipelines, Azure-side integration writing to Dataverse, system-to-system sync
Why It Saves Cost
Application users in Dataverse do not consume user licenses. The Azure Function itself needs Azure consumption cost, not PP license.
Risks
Application user acts with its configured roles — ensure least privilege. No per-row ownership delegation by default.
Governance Note
Source: Create a Dataverse application user
🌎
Pattern 6 — Standardize an Approved Connector Catalog

Publish a list of pre-approved connectors for each environment tier. Standard-only for departmental, premium-allowed only for enterprise environments with licensed users.

When to Use
Any organization with multiple teams building on Power Platform
Why It Saves Cost
Prevents accidental premium connector use in standard-licensed environments. Prevents license non-compliance by design.
Risks
Over-restriction can block legitimate business needs. Catalog needs a review process for exceptions.
Governance Note
Enforce via DLP policies in Power Platform admin center. Document the approved catalog in your CoE guidance.
Section 10

Premium is Justified When…

This section is as important as any cost-saving guidance. A guide that only focuses on cost reduction without acknowledging when premium is the correct choice is not credible — and architects who recommend premium-avoidance in the wrong scenarios create expensive technical debt.

Premium Justified Premium licensing is justified when the alternative would create technical debt, security risk, operational fragility, or ongoing maintenance burden whose total cost exceeds the premium license. "It's cheaper up front" is never a complete cost analysis.

When a solution requires a relational data model with multiple tables, referential integrity, complex queries across entities, field-level security, advanced audit trails, or business rules that execute on data changes — Dataverse is the correct choice. Attempting to replicate this in SharePoint creates a fragile, unsupported workaround with higher maintenance cost than the premium license.

Field-level security (restricting specific columns from specific users), row-level security based on team/business unit hierarchy, and advanced audit logging that cannot be compromised in regulated environments — these require Dataverse. SharePoint item-level permissions are not equivalent for regulated data scenarios.

When a solution handles high concurrent writes (hundreds per minute), requires ACID transactional guarantees, or must handle tens of millions of records with complex filters and sorts — Dataverse or Azure SQL is appropriate. SharePoint's performance at high concurrency is not guaranteed and not the right tool for high-throughput transactional systems.

When integrating with SAP, Salesforce, ServiceNow, or other enterprise SaaS platforms — the premium connector for that platform often provides a battle-tested, Microsoft-maintained integration with robust error handling, authentication, and pagination. Building a workaround via Azure Functions to avoid the premium connector frequently results in significantly more development and maintenance cost.

High-value automation processes (purchase order processing, customer onboarding, financial reconciliation) where reliability, observability, and auditability must be demonstrably production-grade. Per-flow licensing for these scenarios provides a flat, predictable cost for the automation regardless of user count, and justifies the investment vs per-user at scale.

A SharePoint-based workaround maintained by developers costs developer time every time SharePoint changes, every time the list schema needs to evolve, and every time a relational query grows more complex. After 18 months of maintenance cost on a "free" SharePoint solution, the premium Dataverse license may be cheaper in total cost. Always model the 3-year TCO, not just the year-one license cost.

Section 11

Anti-Patterns, False Savings & Red Flags

These are patterns that appear to save cost but create operational risk, security liability, or technical debt that outweighs the licensing saving. An architect who recommends these patterns is not saving the client money — they are deferring cost to a harder-to-recover-from future problem.

Anti-Pattern Excel as a Production Database. No concurrent write protection. No transactional integrity. No audit trail. No conflict resolution. Files become corrupted under simultaneous editing. Absolute ceiling: proof-of-concept and demos only. Never in production, regardless of the licensing saving. This is the most common false economy in the Power Platform ecosystem.
Anti-Pattern Unsupported Tricks to Avoid Premium Licensing. Building workarounds specifically designed to avoid triggering premium licensing requirements — such as chaining flows in unsupported ways, using internal connector APIs not exposed in the official connector — is not cost optimization. It is license avoidance and violates Microsoft's product terms. These patterns break without warning on product updates and are not supportable.
Anti-Pattern Building a Custom Connector to Avoid a Premium Connector. Custom connectors are also premium. This achieves nothing on the licensing front while adding a connector maintenance burden. The only reason to build a custom connector is API contract control — not licensing savings.
Anti-Pattern Using Personal User Accounts for Critical Production Automation. When the flow owner's account is disabled, the flow stops. In production this is a P1 incident. Service principals exist precisely to solve this problem.
Anti-Pattern No Environment Strategy. Every project in the default environment. No dev/test/prod separation. No solution management. When something breaks in production, nobody knows what changed or when. The "cost saving" of not setting up environments is routinely paid back in incident recovery costs.
Anti-Pattern Buying Per-User Premium for Everyone Without Usage Analysis. Before purchasing per-user for your entire organization, map which users actually need premium capabilities. Users who only access M365-backed standard apps may not need premium at all. Users who need only one premium app may be better served by per-app licensing. This analysis takes hours and can save thousands per year.
Anti-Pattern Unmanaged Solutions in Production. Solutions not in a managed deployment are mutable by any maker in the environment. Changes bypass ALM, bypass testing, and bypass approval gates. This is not a licensing issue but a governance issue that directly multiplies operational cost through incident frequency.
Anti-Pattern Storing Secrets in Flow Variables or SharePoint Lists. Client IDs, client secrets, API keys in plain text in flow configuration or SharePoint lists. Use Azure Key Vault (accessible via custom connector or Azure Function) or environment variables with secret handling. The security remediation of a secret leak costs far more than the infrastructure for proper secret management.
False Economy Cheap Today, Expensive Tomorrow. The most insidious cost pattern: a solution that costs less up front but accumulates a growing maintenance, rework, and incident-response burden. Common examples: SharePoint-based relational data that outgrows its design, service account flows that break at every offboarding cycle, unmanaged environments that proliferate until storage costs exceed the Dataverse license that would have governed them.

Checklists

✓ Licensing Review Checklist

  • Map every connector used to its tier (standard / premium / custom) using the official connector reference lists
  • Confirm that all users interacting with premium flows/apps have the required license type
  • Evaluate per-app vs per-user: calculate the break-even at current pricing for your user counts
  • Identify any flows using HTTP connector — evaluate per-flow license vs per-user
  • Confirm Managed Environment license coverage before enabling Managed Environments
  • Validate Dataverse storage allocation is sufficient for planned environments
  • Check if any Dynamics 365 licenses already in use cover Power Platform entitlements

✓ Connector Selection Checklist

  • Identify the data source and target for each integration point
  • Check the standard connector list first — use standard if available
  • If premium connector seems needed, ask: can an Azure Function or Logic App sit in front of this to avoid the connector?
  • Document connector selection rationale at design review ("standard evaluated, premium justified because...")
  • Add connector selection to DLP policy review for the target environment
  • Never use a custom connector to circumvent premium licensing — it is still premium

✓ Environment Setup Checklist

  • Define environment strategy: how many environments, what purpose, what solutions live where
  • Confirm each environment has a named owner and a defined lifecycle
  • Apply DLP policies at environment creation, not as an afterthought
  • Confirm Dataverse database is only provisioned where actually required (storage cost)
  • Document which environments are Managed Environments and confirm license coverage
  • Set environment-level admin access; restrict default environment from production use

✓ Service Identity Checklist

  • Identify all production flows and their current owners
  • Flag any flows owned by named users that are business-critical or unattended
  • Create service principals for critical automation flows
  • Create application users in Dataverse for backend integration (non-interactive)
  • Assign least-privilege roles to all service principals and application users
  • Store client secrets / certificates in Azure Key Vault; never in flow variables or lists
  • Document service principal IDs and role assignments in your solution documentation

✓ Pre-Client Proposal Checklist

  • Have you mapped each solution component to its required license tier?
  • Have you evaluated per-app vs per-user for the proposed user counts?
  • Have you documented why Dataverse is or isn't required?
  • Have you validated connector choices against the official standard/premium connector lists?
  • Have you identified the identity strategy for all automation components?
  • Have you included a 3-year TCO estimate, not just year-one license cost?
  • Does the proposal include an environment and ALM strategy?

Scenario 1 — Small Internal Team App

Requirement A 15-person facilities team needs an app to log maintenance requests, assign them to technicians, and track completion.
Low-Cost OptionCanvas App + SharePoint List + Power Automate (standard connectors) + Teams notifications. Estimated additional license cost: £0 (M365 included). Build time: 3–5 days.

RecommendationStart with SharePoint. Ship fast. Revisit in 6 months if requirements grow. Do not pre-build complexity.

Scenario 2 — Departmental Approval Workflow

Requirement50-person HR department needs a leave approval workflow with manager email approval and SharePoint audit log.
Low-Cost OptionPower Automate + SharePoint (list for leave records) + Approvals connector + Outlook connector. All standard. Zero additional licensing for M365 users.
When Premium Becomes NecessaryIf integration with SAP HR or Workday is required for leave balance sync → premium connector or custom connector required. Per-flow license likely most economical.

Scenario 3 — Enterprise Solution with External API Integration

Requirement200-user customer service team needs a Canvas App with Dataverse backend, integrating with Salesforce for customer data and ServiceNow for ticketing.
Recommended ArchitectureCanvas App + Dataverse + Salesforce connector (premium) + ServiceNow connector (premium). Per-user premium for all 200 users. Service principal owns integration flows.
Risk NotesDataverse justified by relational data model and field-level security. Premium connectors justified by enterprise integration complexity. Workarounds would cost more. 3-year TCO favors premium.
Premium JustifiedThis scenario genuinely requires premium. Trying to avoid it via custom connectors or SharePoint workarounds adds months of development and fragile maintenance burden.

Scenario 4 — Regulated Environment

RequirementFinancial services organization, 100 users, app handling customer financial data. Must meet data residency, audit, and access control requirements.
Required ArchitectureManaged Environment, Dataverse with field-level security and audit enabled, service principal for all automation, IP firewall, DLP policies enforced, solutions only via managed deployment. Premium required for all users.
Cost Optimization Within PremiumEvaluate per-app vs per-user for users accessing only one app. Evaluate per-flow for unattended automation flows. Confirm D365 license entitlements before buying standalone Power Apps licenses.

Scenario 5 — M365-First Low-Cost Alternative

RequirementDepartment wants to replace a spreadsheet-based tracker for 30 people. Basic CRUD, view all records, simple filter.
Zero-Cost PathMicrosoft Lists (Teams-integrated) + Power Automate (standard connectors for notifications). Cost: £0. Build: 1 day. Supports 30 concurrent users easily at this data volume.
Common Mistake to AvoidBuilding a Canvas App backed by Dataverse for this requirement. Adds £2,220–£8,880+ per year in licenses for functionality Microsoft Lists provides for free.

Section 18

Glossary

TermDefinition
Standard ConnectorA connector that is included in Microsoft 365 and base Power Platform license entitlements. Using only standard connectors in a flow or app does not require a premium license. Examples: SharePoint, Teams, Outlook, Planner, Forms, OneDrive, Approvals.
Premium ConnectorA connector that requires a standalone Power Apps Premium, Power Automate Premium, per-app, or per-flow license. A single premium connector in a flow requires premium licensing for all users of that flow. Examples: SQL Server, Dataverse, HTTP, Salesforce, ServiceNow.
Custom ConnectorA connector built by you to connect Power Platform to an API not covered by built-in connectors. Treated as premium for licensing purposes regardless of what API it connects to.
DataverseMicrosoft's enterprise relational database platform built into the Power Platform. Provides relational data model, row- and field-level security, business rules, plugins, and advanced audit. Requires premium licensing for all users of any app or flow that accesses it via the Dataverse connector.
Managed EnvironmentA Power Platform environment with additional governance controls enabled: solution checker enforcement, sharing controls, IP firewall, weekly digest, usage insights. Requires all active users to have a qualifying premium license or Managed Environments add-on.
Power Platform PipelinesA built-in ALM tool within Power Platform for promoting solutions between environments (dev → test → production). Requires a Dataverse database in the host environment. Best suited for maker-led teams with no code components.
Application UserA non-interactive user identity in Dataverse mapped to an Azure AD service principal. Used for backend system-to-Dataverse integration without consuming a user license. Does not require a Power Apps license.
Service PrincipalAn Azure Active Directory application identity (not tied to any human user) used for machine-to-machine authentication. Can own Power Automate flows without a per-user license. Recommended for critical production automation.
Managed IdentityAn automatically managed AAD identity attached to an Azure resource (Function App, Logic App, VM, etc.). Credentials are managed by Azure — no secrets to rotate. Preferred identity for Azure-side integrations calling into Power Platform or Dataverse.
Service AccountA shared named AAD user account (not tied to a person) used for automation. Requires a full user license (including Power Platform entitlements). Acceptable as a short-term workaround; service principals are the recommended long-term replacement due to better security and no license dependency.
Per-App LicenseA Power Apps license that grants a user access to one specific app. More cost-effective than per-user when a user needs access to only one or two apps. Typically ~3–5× cheaper per user per app than per-user, at low app counts.
Per-User LicenseA Power Apps or Power Automate license that grants a user unlimited access to all apps/flows (within the plan). More cost-effective than per-app when a user accesses three or more apps. Validate the break-even point with current pricing.
Per-Flow LicenseA Power Automate license that covers a specific number of flows (typically 5) at a flat monthly cost, regardless of how many users trigger them. Often the most economical option for high-user-count, low-flow-count automation scenarios — e.g., 200 users triggering 2 shared premium flows.
