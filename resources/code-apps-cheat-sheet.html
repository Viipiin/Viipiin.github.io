overview What are Code Apps & when to choose them Source: learn.microsoft.com/power-apps/developer/code-apps/overview
Code Apps let pro-developers build Power Platform apps using any modern web stack (React/Vue/Angular/Svelte) authored in VS Code, packaged with Vite, and run inside the Power Apps player. Apps inherit Power Platform identity (Entra ID), connectors, governance (DLP, tenant policies), environments and sharing. Choose Code Apps when you need: complex UI (charts, maps, custom layouts), heavy client logic, npm libraries, true unit testing, Git workflows, or pixel-perfect design that Canvas can't easily match. Choose Canvas instead when makers need to compose with formulas. Choose Model-driven when Dataverse + standard pages are enough.

auth Identity, PowerProvider & Initialize
IdentityApps run as the signed-in user via Microsoft Entra ID — no token plumbing needed
SDK entry-pointimport { initialize } from "@microsoft/power-apps/app";
When to initCall initialize() exactly once at app start (in PowerProvider.tsx) before any data call
User contextconst user = await getCurrentUser(); — name, email, tenantId, oid
Connection authAll connector calls use the connection-reference's stored credentials (no client secrets in code)
Sign-in UXPower Apps host shell handles SSO; local dev uses pac code run tunnel for connection brokering
Roles & sharingUse Power Apps sharing UI; honor environment security roles for Dataverse
// src/PowerProvider.tsx — minimal pattern import { initialize } from "@microsoft/power-apps/app"; import { useEffect, useState } from "react"; export function PowerProvider({ children }: { children: React.ReactNode }) { const [ready, setReady] = useState(false); useEffect(() => { initialize().then(() => setReady(true)); }, []); if (!ready) return <div>Loading…</div>; return <>{children}</>; }

add-data-source Wire a Connector — Step by Step
1. List referencespac code list-connection-references  — get the connection-id
2. List datasets (if any)pac code list-datasets -a <connector-name>
3. List tables / sprocspac code list-tables -a <connector> -d <dataset>
4. Add itpac code add-data-source -a <connector> -c <connection-id> -d <dataset> -t <table>
5. OutputAuto-generated typed client under src/Services/<ConnectorName>Service.ts + DTOs in src/Models/
6. Use itImport the service and call typed methods — no boilerplate auth code
// Example: add Dataverse "accounts" table pac code add-data-source \ -a "shared_commondataservicepowerapps" \ -c "<connection-reference-id>" \ -t "accounts" // Example: add SQL stored procedure pac code add-data-source \ -a "shared_sql" \ -c "<connection-reference-id>" \ -d "default,server.database.windows.net,mydb" \ -sp "dbo.GetTopOrders"

dev Local Run & Debug
  • Open 2 terminals in VS Code.
  • Terminal 1 — Power tunnel: pac code run — brokers connector traffic via your auth profile.
  • Terminal 2 — Vite: npm run dev — opens http://localhost:5173.
  • Hot-reload: edits to .tsx/.css apply instantly.
  • Console & network: Edge/Chrome DevTools. Network shows tunnel requests routed via localhost.
  • Stop with Ctrl + C in both tabs.
  • Restart the tunnel any time you switch pac auth select.
💡 If a connector returns 401/403 locally, re-run pac auth create and restart pac code run.
deploy Build & Publish
  1. Ensure pac auth who points at the target environment.
  2. npm run build — Vite emits dist/.
  3. pac code push — uploads the bundle & registers a new app version.
  4. App appears in Power Apps maker portal under Apps → Code apps.
  5. Share like any Power App: Share → assign users/security groups.
  6. For ALM, add the code app to a solution and use pipelines/PAC build tools.
  7. Subsequent pac code push = new published version (auto-bumped).
🚀 First push prompts for app display name & description. Re-running push updates the existing app — no second prompt.

ALM Solutions, Pipelines, Source
  • Add the published code app to an unmanaged solution in the maker portal.
  • Include the connection-references it depends on.
  • Export managed/unmanaged via pac solution export.
  • Use Power Platform Pipelines or Azure DevOps build tools for promotion.
  • Keep power.config.json in Git; environment IDs differ per stage.
  • Tag releases in Git matching pac code push versions.
  • Don't check in node_modules/, dist/, or .env.local.
limits Known Limitations Preview
  • Preview: subject to breaking changes — pin SDK versions.
  • Requires Premium / per-app Power Apps license.
  • Not available in GCC High / DoD clouds at preview time.
  • No offline mode (yet).
  • Mobile native shell support limited — design responsively for web.
  • Bundle size cap: keep dist/ reasonably lean for fast load.
  • Cannot directly embed in Model-driven forms (use iframe or PCF).
  • No Power Fx in code — call connectors via the SDK instead.
  • Service principal auth for the dev loop is not supported.

troubleshoot Common Errors & Fixes
"pac code not recognized"Update PAC: pac install latest. Restart terminal.
401 on connector calls (local)Tunnel auth expired. Stop pac code run, run pac auth create, restart tunnel.
Blank page after pushForgot base: "./" in vite.config.ts. Rebuild and push.
"PowerSDK undefined"You called a service before initialize() resolved. Wrap with PowerProvider.
Types out of syncRe-run pac code add-data-source for that connector.
"Connection reference not found"You're on the wrong env. pac auth who → switch with pac auth select.
"App not visible in maker portal"Push completed but you filtered Code apps off. Maker → Apps → filter "Code apps".
Slow first load in productionBundle too big. Run vite build --report, code-split heavy routes.
CORS errors locallyAlways call connectors through generated services — never raw fetch to a connector URL.
"Premium connector blocked"User lacks Premium licence OR DLP policy blocks the connector class.
"Solution import fails"Connection references unmapped. Map them in the solution import wizard.
HMR not refreshingCheck for syntax errors in console; restart npm run dev.
license Governance Cheats
  • Maker: Premium Power Apps per-user.
  • User: Premium per-user OR per-app plan covering the published app.
  • Admin enable: Power Platform Admin Center → Settings → Code apps → On.
  • DLP: Each connector in your code app is governed by the env's DLP policy.
  • Audit: Code app activity flows through Power Platform analytics + Purview.
  • Capacity: Bundle size + connector calls count toward env quotas.
  • Sharing: Use Entra security groups for scale; avoid per-user grants.

FAQ Quick Answers
Do I need Dataverse?No. You can use SharePoint, SQL, or any connector — Dataverse is just one option.
Can I use Next.js / Remix?Code Apps target a static client bundle; SSR frameworks aren't supported. Use Vite + React/Vue/Svelte/Angular SPA mode.
Can I use Tailwind CSS?Yes — install tailwindcss, init config, import in index.css. Works seamlessly with Vite.
Call Azure Functions / APIs directly?Yes — via a custom connector wrapper. Direct fetch works for public APIs, but loses governance.
Embed in a model-driven page?Use an iframe with the code app's URL, or embed via a PCF host control.
How do I version & rollback?Each pac code push creates a new version; previous versions are retained & restorable from the maker portal.
Does it work on mobile?Yes, in the Power Apps mobile player — design responsively. Native shell APIs (camera, GPS) limited via SDK; use the device's web APIs or connectors.
Users without licenses?No preview without licensing — Premium licensing applies to all runtime users.
Is the source code visible?The compiled JS bundle is downloaded to the client (like any web app). Keep secrets server-side or in connector auth.
Migrate from Canvas?No automatic migration. Rebuild the UI in React; reuse connectors via pac code add-data-source.
